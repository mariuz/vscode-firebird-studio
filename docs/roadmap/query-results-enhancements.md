# Query Results Grid Enhancements

**Inspired by**: [vscode-mssql](https://github.com/microsoft/vscode-mssql)'s ongoing Query Results Grid work — Text View mode for result sets (1.35.0), selection aggregations (count/sum/average shown when selecting rows or columns, 1.37.0), configurable grid font via `mssql.resultsFontSize`/`mssql.resultsFontFamily` (1.28.0), and a newer results-grid rework adding per-column show/hide alongside freeze (1.44.0, preview).

## Current state in Firebird Studio

**All five phases are done** (phase 3 turned out to already exist — see below). This item is complete. The grid this extends (`src/result-view/htmlContent/js/app.js`, backing `ResultView`) was already fairly mature going in — worth being precise about what was already there so this doc only proposed genuinely new ground:

- Already done (before this doc): sort/filter per column (DataTables-backed), CSV/JSON/XLSX/PDF export, pagination, one tab per statement in a multi-statement batch (`$tabBar`/`.fb-tab` — Firebird Studio's existing answer to "multiple result sets," a different but equivalent shape to mssql's single-query-multiple-result-set model), inline row editing with a script preview (add/edit/delete, `$apply`), and a **Freeze Column** toggle (`$freezeToggle` in `app.js`).
- **Phase 3 — Column show/hide — turned out to already be done, missed during this doc's original survey.** `buildExportButtons()`'s `{ extend: "colvis", text: "Columns", columns: ":gt(0)" }` button (present before this doc was even written) is DataTables' own built-in "Buttons - Column visibility" extension (`b-colvis-1.5.4`, bundled in the vendored `datatables.min.js`) — a checkbox-list toolbar button hiding/showing columns via `column().visible()`, excluding only the leading row-actions column, rendered via the DataTable's existing `dom: "Bfrtip"` Buttons toolbar. Exactly the feature this phase proposed, already shipped; no code changes were needed once this was noticed — this section is purely correcting the record.
- **Phase 1 — Text View mode — done.** A "📄 Text View" toolbar button next to the existing grid toolbar (`$textViewToggle`, alongside `$freezeToggle`/`$chartToggle`) toggles a panel below the table rendering the *original* headers/rows (not DataTables' current sort/filter/page state — the same deliberate scope cut the pre-existing chart panel already established) as tab-client-style aligned plain text, via a new pure `buildTextView(headers, rows)` function: a header line, a dashed separator line, then one line per row, every column padded to its widest value (header included), `NULL`/`undefined` cells rendered as the literal `NULL` (distinguishing them from an empty string, which renders as blank). A "Copy" button in the panel copies the rendered text to the clipboard via the existing `copyToClipboard()` helper. Rendered lazily (only on first toggle-open, matching the chart/plan panels' existing pattern), not on every render of the table.
- **Phase 2 — Selection aggregations — done, with one deliberate deviation from the original proposal.** The proposal above said "sum/average/min/max for the selection when every selected cell **in a column** parses as numeric" — implemented instead as an aggregate over whatever numeric-looking cells are anywhere in the selection (not requiring column-wide purity first), closer to how a spreadsheet's own selection status bar behaves and more useful for a selection spanning several mixed-type columns; a stricter per-column requirement would silently show nothing for the common case of a range selection that happens to include one non-numeric column. A `$selectionStats` readout in the toolbar (next to `$status`) updates on every selection change (wired into `highlightSelection()`, the single function every mousedown/mouseenter selection-drag call already funnels through) via new pure `computeSelectionStats(rows)`/`formatSelectionStats(stats)` functions — `Count: N` always, plus `Sum`/`Avg`/`Min`/`Max` (each rounded to 2 decimals) when at least one selected cell parses as numeric. Cleared (along with the selection itself) by the existing `clearSelection()`.
- **Phase 4 — Grid font customization — done.** New `firebird.resultsFontSize` (number, px, `0` = unset) and `firebird.resultsFontFamily` (string, `""` = unset) settings, mirroring mssql's naming exactly. Threaded from `getOptions()` into `ResultView`'s existing "getData" response payload (`src/result-view/index.ts`, alongside the pre-existing `shortcuts` field — both batch and single-result-set branches), read by the webview's `batchData`/`message` handlers and applied via a new `applyResultsFont(fontSize, fontFamily)`, which sets `--fb-results-font-size`/`--fb-results-font-family` CSS custom properties on `document.documentElement`. `app.css`'s `table.dataTable` rule reads them with a `var(..., 12px)`/`var(..., inherit)` fallback, so every batch panel's `<table>` (each one rebuilt per query run) picks the setting up automatically without needing re-application per table — only the one root-level property set matters. The pure half (`resultsFontProperties(fontSize, fontFamily)`, computing which properties to set) is separated from the DOM-touching half (`applyResultsFont()`, which just applies whatever that function returns) the same way `computeSelectionStats()`/`formatSelectionStats()` split above already does.
- **Phase 5 — "View Table Diagram" — done.** A "🗺 View Table Diagram" toolbar button next to `$freezeToggle` posts a `viewTableDiagram` message with the table name currently entered for row editing (`$tableNameInput.val()`; a client-side guard shows "Enter a table name first" via `$status` if it's empty, rather than sending an unusable request). `ResultView.handleMessage()` delegates it to extension.ts via the same `EventEmitter`-based pattern already established for `analyzeResults`/`analyzePlan` (`ResultView` has no direct dependency on `SchemaDesigner`, matching the existing "cross-feature wiring lives in extension.ts" convention) — the handler resolves the actual connection from `Global.activeConnection` (the same implicit source row editing's own `applyChanges()` already resolves its connection from, via `Driver.runQuery()` with no explicit `connectionOptions`; shows a "No Firebird database selected!" error if unset) and calls the *pre-existing* `SchemaDesigner.openForAlterTable(dbDetails, tableName)` — the exact method `NodeTable.alterTable()` already uses for the tree's own "Alter Table" action, confirming the doc's original assessment that no new schema-introspection logic was needed here.

### Testing

`src/test/result-view-webview.test.ts` gained a `buildTextView()` suite (6 tests), a `computeSelectionStats()`/`formatSelectionStats()` suite (9 tests), and a `resultsFontProperties()` suite (4 tests), all via the file's existing `__test__` hook pattern: header/separator/row-line shape and column-width padding for Text View; all-numeric, mixed numeric/text, all-text, empty, negative/decimal, and whitespace-padded-cell cases for the aggregation functions; unset/size-only/family-only/both cases for the font-properties function. The DOM wiring itself (button/readout creation, `highlightSelection()`'s hook into it, `applyResultsFont()`'s `document.documentElement.style.setProperty()` calls) isn't independently tested, consistent with this repo's established boundary for webview inline JS — every new *pure* function here falls into the same "pure, no DOM/jQuery" category this file's own section comments already carve out as testable, the same category `sqlLiteral()`/the chart SVG builders fall into. The two new `config.ts` accessors (`_resultsFontSize()`/`_resultsFontFamily()`) also have no dedicated unit test, consistent with every other accessor in that file — `config.ts` has no test file at all today, and these two don't establish a new pattern by being the first to skip one.

Phase 5's extension-host-side wiring is different in kind from the other four phases — a real message-delegation chain, not a pure data transform — so it gets suite-tier coverage instead: new `src/test/suite/result-view-integration.test.ts` drives the real `ResultView.handleMessage()` (no live database connection needed, matching `query-plan-view-integration.test.ts`'s precedent for `QueryPlanView`'s own synchronous "analyzePlan" emission test) and confirms `viewTableDiagram` fires with the table name unchanged, including for two different tables in sequence. What's *not* covered by an automated test: `SchemaDesigner.openForAlterTable()` actually being called with the right arguments from extension.ts's handler (that handler lives in `extension.ts#activate()` itself, which isn't independently unit- or suite-testable in this codebase — the same boundary every other `commands.registerCommand()` callback in that file already sits behind) and the real webview's "Enter a table name first" guard (client-side, same "webview inline JS DOM wiring isn't independently tested" boundary as everything else in this doc).
