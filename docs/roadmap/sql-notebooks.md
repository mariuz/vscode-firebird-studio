# SQL Notebooks

**Inspired by**: [vscode-mssql](https://github.com/microsoft/vscode-mssql)'s SQL Notebooks ("Jupyter-based SQL notebooks with rich results and multi-kernel support").

## Current state in Firebird Studio

**Phases 1 and 3 are done.** `.fbnb` notebooks (`contributes.notebooks`, `firebird.notebook.new` to create one) run Firebird SQL cells via VS Code's native Notebook API:

- `src/sql-notebook/serializer.ts` (`FirebirdNotebookSerializer`) ‚Äî round-trips a small custom JSON shape (`{cells: [{kind, value, languageId}], metadata?: {connectionId}}`), not Jupyter's `.ipynb` nbformat ‚Äî this is a single-kernel, SQL-only notebook, so nbformat's cross-kernel metadata would be unused ceremony. A brand-new/empty file opens with one blank SQL cell.
- `src/sql-notebook/controller.ts` (`createSqlNotebookController()`) ‚Äî a `NotebookController` (`supportedLanguages: ['sql']`) whose `executeHandler` runs each cell's text through `Driver.runBatch()` (the same call the regular editor/"Run Query" flow uses) and renders each `BatchResult` as its own `NotebookCellOutput`: an error renders via `NotebookCellOutputItem.error()`, a DDL/DML success message as plain text, and rows as a GitHub-flavored markdown table (`src/shared/notebook-render.ts#renderRowsAsMarkdown()`, unit-tested) ‚Äî markdown is one of VS Code's natively-rendered output mime types, so Phase 1 needs no custom notebook renderer yet.
- **Connection binding is now persisted** (`resolveNotebookConnection()` in `controller.ts`): the first cell run without a bound connection shows the same connection quick-pick pattern used elsewhere (`schemaDiff`'s source/target picker); the chosen connection's *id* (never its password) is then written into the notebook's own metadata via `NotebookEdit.updateNotebookMetadata()` + `workspace.applyEdit()`, and `FirebirdNotebookSerializer` round-trips that metadata to/from the `.fbnb` file's own `metadata.connectionId` key. Reopening a notebook (or restarting VS Code) now resolves straight from that persisted id instead of re-prompting ‚Äî falling back to the picker again if the id no longer matches any saved connection (e.g. it was since removed). The resolved *password* is still never persisted anywhere on disk; it's re-resolved via `CredentialStore`/`Driver.resolvePassword()` each session and kept only in the existing in-memory `boundConnections` map (module-level, keyed by `notebook.uri.toString()`), which is what actually carries it once resolved.

The pre-existing state this replaces: query execution was previously file/selection based only (`Driver.runQuery()`/`runBatch()` against a `.sql` document), with results always going to the `src/result-view/` webview ‚Äî there was no notebook surface and no `NotebookController` registration in `src/extension.ts`.

### Phase 3 ‚Äî persisted connection binding

- **Verified in a real Extension Development Host** (`npm run test:vscode-host`), not the unit-test tier ‚Äî `NotebookController`/`NotebookSerializer`/`NotebookCellExecution` still aren't mocked in `src/test/mocks/vscode.ts` (see below), so this is the only tier where the real Notebook API is available at all. A new `src/test/suite/sql-notebook-integration.test.ts` covers: `FirebirdNotebookSerializer` round-trips `metadata.connectionId` through a real `serializeNotebook`/`deserializeNotebook` pair (and omits the `metadata` key entirely when unset, rather than writing `{connectionId: undefined}`); `resolveNotebookConnection()` (exported for this) resolves straight from a persisted id without ever touching `window.showQuickPick`, when that connection still exists; and it falls through cleanly (returns `undefined`, doesn't throw) when the persisted id points at a connection that's since been removed and there's nothing else to fall back to. Deliberately never drives the real picker in these tests ‚Äî there's no UI to answer it in a test host, and every scenario tested resolves (or correctly gives up) before the picker would open.
- **Found and fixed a real, unrelated latent bug along the way**: opening a notebook whose cells have `languageId: "sql"` triggers the SQL completion provider (`src/language-server/db-words.provider.ts#KeywordsDb.getSchema()`), which ‚Äî with no active connection yet set (true for a freshly-opened Extension Development Host) and the default `firebird.codeCompletion.database` setting (`true`) ‚Äî called `this.build(Global.activeConnection, ...)` with `Global.activeConnection` still `undefined`, throwing inside `build()` on `conOptions.database`. The `try/catch` around it swallowed the throw and logged it, so this never broke the UI, but it did mean completions silently produced nothing and the setting's own intent (only query the database for completions when it's enabled) was already broken independently of the crash: the old condition (`!Global.activeConnection && !codeCompletionDatabase`) only bailed when *both* were true, so a connection-but-setting-disabled case wrongly queried anyway. Fixed to bail whenever *either* is true (`!Global.activeConnection || !getOptions().codeCompletionDatabase`) ‚Äî correct on both counts, and confirmed by the noisy `TypeError: Cannot read properties of undefined (reading 'database')` log lines disappearing from the same suite run once fixed. Predates this phase entirely (last touched in an unrelated earlier commit); surfaced only because this phase's new tests were the first in this repo to open a real `.fbnb` notebook document in the Extension Development Host test tier.

## Proposed feature

VS Code has a first-class Notebook API (`vscode.NotebookController`, `NotebookSerializer`) ‚Äî this doesn't need Jupyter itself, just VS Code's native notebook UI:

1. **Serializer** ‚Äî a `.fbnb` (or reuse `.ipynb` for interop with existing Jupyter tooling) file format: cells are either markdown or Firebird SQL. Implement `NotebookSerializer.deserializeNotebook`/`serializeNotebook` to round-trip cell content + outputs.
2. **Controller** ‚Äî register a `NotebookController` for the `sql`-cell language that, on cell execution, calls `Driver.runQuery()` (same call the regular query flow uses) and renders each `BatchResult` as a notebook cell output ‚Äî reuse the existing result-rendering logic from `src/result-view/` rather than building a second grid renderer; the notebook cell output can embed the same HTML/webview-renderer content as a `NotebookCellOutputItem` of mime type `text/html` (or a custom `application/vnd.firebird.result+json` renderer registered via `contributes.notebookRenderer` for a richer grid than plain HTML allows).
3. **Connection binding** ‚Äî a notebook needs to remember which saved connection it's bound to (metadata stored in the notebook document, similar to how `.sql` editors already track an active connection via `Global`/status bar) ‚Äî first cell run without a bound connection should prompt the same connection quick-pick used elsewhere.
4. **Multi-kernel** mention in mssql's feature table likely refers to supporting both SQL and, e.g., PowerShell/Python kernels in one notebook (for scripting around results) ‚Äî that's out of scope for an initial Firebird port; scope this feature to a single Firebird-SQL kernel and revisit multi-kernel only if there's demand.

## Technical notes

- This is one of the larger lifts on this list: it's a genuinely new UI surface (notebooks), not an extension of an existing webview, and needs its own `contributes.notebooks` + `contributes.notebookRenderer` entries in `package.json`.
- Output rendering is the crux of the "rich results" promise ‚Äî a plain markdown table per cell is easy but loses sort/filter/export; a custom notebook renderer that reuses `src/result-view/htmlContent` is more work but keeps parity with the existing query-results experience users already know.
- Needs its own unit/e2e testing story: `NotebookController`/`NotebookSerializer` aren't in `src/test/mocks/vscode.ts` today and would need mocking (same gap noted for the Copilot chat participant) or coverage pushed entirely to the `suite`/vscode-host tier.
