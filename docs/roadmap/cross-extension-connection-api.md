# Cross-Extension Connection Sharing API

**Inspired by**: [vscode-mssql](https://github.com/microsoft/vscode-mssql)'s connection-sharing permissions API (1.34.0) — "Added connection-sharing permissions API for other VS Code extensions that wish to use MSSQL connections," letting a separate extension (e.g. a data-visualization or ORM-tooling extension) reuse a connection the user already configured in mssql, gated by an explicit per-requesting-extension permission the user grants.

## Current state in Firebird Studio

**All four phases are done.** This item is complete. Before this, `activate()` (`src/extension.ts`) didn't return anything — there was no `exports` object at all, so another VS Code extension had no supported way to discover or reuse a Firebird Studio connection. Anything wanting Firebird access still has to either bundle its own `node-firebird` dependency and ask the user to re-enter connection details, or (the one other existing integration point) go through the MCP server (`docs/roadmap/mcp-server.md`) — but that's scoped to AI-agent hosts speaking MCP-over-stdio to a spawned subprocess, not an in-process API another VS Code extension's own code could call directly and synchronously.

- **Phase 1 — `listConnections`/`getActiveConnection` — done.** `src/connection-sharing/index.ts` exports `listConnections(context, requestingExtensionId?)` (every connection this workspace can see — the same set `FirebirdTreeDataProvider.getHostNodes()` merges: `globalState`-saved connections plus this workspace's own `.vscode/firebird.json`, if any) and `getActiveConnection(requestingExtensionId?)` (whatever's currently in `Global.activeConnection`, or `undefined`). Both reshape a `ConnectionOptions` into a new `SharedConnectionInfo` (`id`/`label`/`host`/`database`/`embedded`) that structurally cannot carry a password — there's no field for one to leak into by accident. Registered as `firebird.connectionSharing.listConnections`/`firebird.connectionSharing.getActiveConnection` commands in `extension.ts`, matching mssql's actual command-based surface exactly as planned below; deliberately **not** declared in `package.json`'s `contributes.commands`, since these are an API surface for other extensions to call via `commands.executeCommand()`, not end-user Command Palette entries. `requestingExtensionId` is accepted and logged (`logger.debug`) but not checked against anything — these two commands were deliberately never gated (see phase 2 below for why), scoped instead to information the tree view already shows visually to anyone looking at this workspace.
- **Phase 2 — permission gate — done.** New `src/connection-sharing/permissions.ts`, `SecretStorage`-persisted (matching mssql's actual `ConnectionSharingService`, not `globalState`), storing a richer `ExtensionGrant` per extension id than originally sketched (`{ read: 'approved' | 'denied', writeEnabled: boolean }` rather than a bare string) so phase 4's write opt-in would have somewhere to live from the start rather than needing a later storage-format migration. `requestConnectionSharingPermission(context, extensionId)` checks a cached grant first; on the first call from a not-yet-seen extension id it prompts once via `showInformationMessage("<id> wants to run read-only queries...", "Approve", "Deny")` and caches the answer — dismissing (Escape/click-away) is deliberately *not* cached, so a dismissed prompt is asked again next time rather than silently treated as a Deny. Only `runQuery`/`runWriteQuery` (phases 3–4) sit behind this gate — `listConnections`/`getActiveConnection` (phase 1) still don't, since the rationale for leaving them ungated (nothing beyond what the tree already shows) hasn't changed. New `firebird.connectionSharing.editPermissions` command ("Review Connection Sharing Permissions...") — the one command in this whole feature that *is* declared in `contributes.commands`, since it's meant for the Command Palette, not another extension — lets the user pick a previously-requesting extension and Revoke (forget the grant, asked fresh next time), Deny future access, or (phase 4) toggle its write access.
- **Phase 3 — `runQuery` — done.** `src/connection-sharing/run-query.ts`'s `runQuery(context, requestingExtensionId, connectionId, sql)`, registered as `firebird.connectionSharing.runQuery` (also not in `contributes.commands`). Checks the phase 2 permission gate, then `validateReadOnlyStatement()` (the exact same function the MCP server's own `run_query` tool already uses — a single read-only `SELECT` only), resolves `connectionId` the same way `listConnections()` does (`globalState` or workspace connections), then calls `Driver.runQuery()` directly. Unlike the MCP server's own `run_query`/`run_write_query` tools, this runs entirely in-process — it's a real VS Code command another extension calls via `commands.executeCommand()`, not code inside a spawned subprocess with no `vscode` API access at all — so, unlike that subprocess, this genuinely can and does go through `Driver`/`CredentialStore` directly rather than needing its own separate credential-passing mechanism. `Driver.runQuery()` already resolves a connection's password internally via `CredentialStore` whenever one isn't already set on the options passed in, so the password never needs to be (and never is) read out into this module at all — only query results ever cross the boundary, exactly as planned. Also awaits the previously-added `Driver.clientReady` first — `activate()` fires `Driver.setClient()` without awaiting it, so `Driver.client` can briefly be undefined right after activation, precisely the scenario another extension calling this command immediately after activation would hit.
- **Phase 4 — opt-in write variant — done** (this phase was explicitly speculative/deferred in the original write-up below; built now on request). `hasWriteAccess()`/`toggleWriteAccess()` (`permissions.ts`) add the separate write opt-in `requestConnectionSharingPermission()` never grants on its own — mirroring `NodeDatabase.toggleMcpWriteAccess()`'s reasoning for the MCP server's own write-query path exactly: a **manual, explicit toggle**, never an automatic first-write-attempt prompt (unlike the base read grant), reached through `editConnectionSharingPermissions()`'s own action menu rather than a separate standalone command (avoiding a second "pick an extension" picker flow) — enabling requires an explicit modal confirmation (`showWarningMessage({modal: true, detail: "..."})`, the same shape MCP's own write-access warning uses) since there's no per-query confirmation once it's on; disabling needs none, matching every other permission-reducing toggle in this codebase. `runWriteQuery()` (`run-query.ts`, registered as `firebird.connectionSharing.runWriteQuery`) checks *both* the read grant and `hasWriteAccess()`, then `validateWriteStatement()` (a single `INSERT`/`UPDATE`/`DELETE` only, the same function the MCP server's `run_write_query` tool already uses) before calling `Driver.runQuery()`. **Deliberately simpler audit trail than the MCP server's own write-query path**: MCP's write-query audit log is a whole file-watcher mechanism specifically because the MCP server runs in a *spawned subprocess* with no way to call back into the extension host directly; `runWriteQuery()` runs in-process, so every write attempt (successful or not) is just logged straight to the Firebird output channel via the existing `logger` — no separate audit-log file or viewer command needed, since the same output channel already serves that purpose without the cross-process complexity MCP had to solve.

### Testing

`src/test/connection-sharing.test.ts` (8 tests) covers the unbundled `listConnections()`/`getActiveConnection()` functions directly against a mock `ExtensionContext`: empty/populated connection lists, multiple connections, the embedded case, and that a password set on the underlying `ConnectionOptions` never appears anywhere in the returned `SharedConnectionInfo`. `src/test/connection-sharing-permissions.test.ts` (18 tests) covers `requestConnectionSharingPermission()`/`hasWriteAccess()`/`toggleWriteAccess()`/`editConnectionSharingPermissions()` by monkey-patching the *mocked* vscode module's `window.showInformationMessage`/`showWarningMessage`/`showQuickPick` — `src/test/setup.ts` redirects every `require('vscode')` call to the exact same cached mock module instance, so mutating it in a test really does affect what the code under test sees; restored after every test. Covers: prompting exactly once per extension id (cached on both Approve and Deny, *not* cached on dismiss), refusing an empty extension id outright, write access requiring prior read approval and its own modal confirmation only when *enabling*, and all three `editConnectionSharingPermissions()` actions (Revoke/Deny/toggle-write). `src/test/connection-sharing-run-query.test.ts` (9 tests) covers `runQuery()`/`runWriteQuery()` the same way, plus a fake in-memory `ClientI` (`Driver.client` assigned directly, no real database) to exercise the actual `Driver.runQuery()` success/failure paths — permission-denied, non-`SELECT`/non-write rejection, unknown connection id, and a real fake-client round trip recording exactly what SQL text was run. `src/test/suite/connection-sharing-integration.test.ts` gained 4 more tests (8 total) driving the *real, bundled* `runQuery`/`runWriteQuery`/`editPermissions` commands via `vscode.commands.executeCommand()`, monkey-patching the *real* `vscode.window.showInformationMessage`/`showWarningMessage` the same way `mcp-server-integration.test.ts` already established for exactly this reason (no person available to click a real dialog's button in a test run) — confirms the real gate refuses a denied extension, reports "connection not found" gracefully for an approved one, refuses a write without a write grant, and that `editPermissions` doesn't hang regardless of whether a grant already exists from an earlier test in the same run.

## Proposed feature (original — see "Current state" above for what shipped and the deviations)

Checked against mssql's actual implementation (`extensions/mssql/src/connectionSharing/connectionSharingService.ts`, not just the changelog blurb) rather than guessing the shape — two design choices there are worth deliberately copying:

- **A command-based surface, not an `activate()` exports object.** mssql registers plain `vscode.commands.registerCommand("mssql.connectionSharing.xxx", (extensionId, ...) => ...)` handlers instead. That's not an arbitrary choice: a VS Code command has no reliable way to know who's calling it, so the *caller* passes its own extension id as the first argument, and the service treats that id as an (unverified, but at least explicit and logged) claim of identity for the permission check below. An `activate()`-returned exports object would have the same "any caller can claim to be anyone" property, so the command-based surface isn't meaningfully less secure — Firebird Studio should follow the same pattern rather than inventing a different one, since a real consumer extension may already assume `commands.executeCommand("firebird.connectionSharing.xxx", ...)`-style discovery if it's used to mssql's shape. Proposed commands: `firebird.connectionSharing.listConnections(extensionId)`, `firebird.connectionSharing.getActiveConnection(extensionId)`, `firebird.connectionSharing.runQuery(extensionId, connectionId, sql)`.
- **Permissions persisted in `SecretStorage`, not `globalState`.** mssql's `ConnectionSharingService` stores its `{ [extensionId]: "approved" | "denied" }` map via `context.secrets.store()`, not `globalState` — worth matching rather than the `globalState`-only approach the rest of this extension's non-credential settings use, since a permission grant is closer in sensitivity to a credential (it's "who gets to run queries as this user") than to a UI preference. First call from an unrecognized extension id triggers `showInformationMessage("<extensionId> wants to use your Firebird connections", "Approve", "Deny")`; the choice is cached, so it's asked once per requesting extension, not per call. A `firebird.connectionSharing.editPermissions` command (QuickPick over `vscode.extensions.all`, matching mssql's own `editConnectionSharingPermissions`) lets the user review or revoke a grant later.
- **Passwords never cross this boundary at all**, the same rule already established for the MCP server's `get_schema` (`docs/roadmap/mcp-server.md`'s point 3) — `runQuery()` executes on Firebird Studio's own side using its own already-resolved `CredentialStore` password; the calling extension never receives connection credentials, only query results.
- Read-only by default (`runQuery()` rejects non-`SELECT` statements via the same `validateReadOnlyStatement()` the MCP server's `run_query` tool already uses) — a write-capable variant, if ever added, should require the same explicit opt-in toggle pattern `docs/roadmap/mcp-server.md` established for MCP write access, not ship enabled by the base permission grant.

## Technical notes

- This was a genuinely different risk profile from anything else in this extension's roadmap: it's the first time *arbitrary third-party extension code*, not a spawned subprocess or a webview's sandboxed JS, gets to call into `Driver`. The permission-gate design above exists specifically to keep that from being an ambient trust grant — nothing beyond `listConnections`/`getActiveConnection` (phase 1, already visible in the tree) works without an explicit per-extension Approve, and nothing write-capable works without a *second*, separately-confirmed opt-in on top of that.
- This shipped without a concrete consumer extension in mind, unlike the "build against a real need" preference this note originally argued for — done at explicit user request once phases 2–4 were confirmed wanted, despite the speculative-API-surface concern still being valid in the abstract. Worth keeping in mind if this surface goes unused in practice: the design is sound and tested, but its actual value depends on some other extension someday calling it.
