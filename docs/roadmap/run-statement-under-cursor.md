# Run Statement Under Cursor

**Inspired by**: [vscode-pgsql](https://github.com/microsoft/vscode-pgsql) (1.9.0) — "Keybinding for 'Run Current Statement' in the Query Editor (default: `Ctrl+Shift+Enter`) executes the statement under the cursor without requiring a selection."

## Current state in Firebird Studio

**Done, all three phases.**

- **Phase 1 — offsets.** New `splitStatementsWithOffsets(sql): SqlStatementRange[]` (`src/shared/sql-splitter.ts`) shares the exact same character-by-character boundary-detection loop as `splitStatements()` — string literals, comments, `SET TERM`, PSQL `BEGIN`/`CASE`/`END` nesting — by tracking a `stmtStart` offset alongside the existing `current` accumulator and computing each flushed statement's trimmed `[start, end)` range from it (`sql.slice(start, end) === text`, verified directly by a dedicated test). `splitStatements()` itself is now a one-line wrapper (`splitStatementsWithOffsets(sql).map(s => s.text)`) — no behavior change, no second implementation of the hard part to drift out of sync.
- **Phase 2 — command.** New `firebird.runCurrentStatement` command (`src/extension.ts`): with an empty selection (just a cursor), finds the statement whose range contains the cursor's document offset (`editor.document.offsetAt(editor.selection.active)`) and runs *only* that statement's text through the existing `Driver.runQuery(sql)` — no changes to execution itself. A cursor sitting in pure inter-statement whitespace (no range contains it, by design — see `splitStatementsWithOffsets()`'s own doc comment) or an actual non-empty selection both leave `sql` as `undefined`, deferring entirely to `Driver.runQuery()`'s own pre-existing selection-aware/whole-document default rather than this command re-implementing that fallback itself. Error handling (the `{notify, message, options}` shape `Driver.runQuery()` rejects with for "no active database"/"no SQL document") mirrors `firebird.runQuery`'s own catch block for a consistent UX between the two commands.
- **Phase 3 — keybinding.** `ctrl+shift+enter` (`cmd+shift+enter` on macOS) in `contributes.keybindings`, scoped `editorTextFocus && editorLangId == sql` (matching this codebase's own existing keybinding-scoping convention for SQL-editor-only bindings, e.g. `firebird.explainPlan`) — matches pgsql's own choice.

## Proposed feature (original)

- Extend (or add a sibling to) `splitStatements()` returning each statement's start/end offset in the original source alongside its text — reusing its existing boundary-detection logic (the genuinely hard part: quoted strings, comments, `SET TERM`) rather than re-implementing statement-splitting a second time just to get positions.
- A new `firebird.runCurrentStatement` command: finds the statement whose `[start, end)` range contains the cursor's document offset, and runs just that one statement through the existing `Driver.runQuery(sql, ...)` path — no changes needed to execution itself, only to *what SQL text gets selected* before the existing call.
- Default keybinding, matching pgsql's own choice for muscle-memory-consistency with a Postgres background: `ctrl+shift+enter` (`cmd+shift+enter` on macOS) — needs to be scoped to `editorTextFocus && resourceLangId == sql` in `package.json`'s `contributes.keybindings` so it doesn't shadow an unrelated global keybinding elsewhere.
- Cursor between statements (in whitespace, or a comment before/after any statement) — fall back to the existing "no selection → whole document" behavior rather than erroring, since the ambiguity has an obvious existing answer already.

### Testing

`src/test/sql-splitter.test.ts` gained a dedicated `splitStatementsWithOffsets()` suite (9 tests): empty/whitespace/lone-semicolon input, every range's `sql.slice(start, end) === text` invariant, a single untermin­ated statement's `[0, length)` range, leading/trailing whitespace excluded from a range, a cursor offset in the pure-whitespace gap between two statements matching no range, a cursor offset correctly resolving to the *second* statement (not the first) in a multi-statement document, a cursor offset inside a `SET TERM`-delimited procedure body resolving to the whole procedure as one range, the `SET TERM` directive lines themselves never appearing in any range, and `splitStatements()`/`splitStatementsWithOffsets()` agreeing on the same statement text in the same order for a real mixed DML+procedure+DML document. `firebird.runCurrentStatement`'s own command body (cursor/selection handling, error display) is VS Code editor/dialog orchestration and isn't independently unit- or suite-tested, matching this codebase's established boundary — it was verified by reading through the exact same `Driver.runQuery()`/error-catch pattern already proven live by `firebird.runParameterizedQuery`'s and `firebird.runQuery`'s own suite coverage, reusing rather than duplicating that path.
