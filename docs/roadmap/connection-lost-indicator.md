# Connection Lost Indicator

**Inspired by**: [vscode-pgsql](https://github.com/microsoft/vscode-pgsql) (1.15.0) — "UI elements in Object Explorer and Query Editor will indicate when an active connection is lost," rather than only surfacing a dropped connection the next time a query happens to fail against it.

## Current state in Firebird Studio

**Done, all three phases.** A connection dropping mid-session (server restart, network blip) is now surfaced immediately instead of only showing up as a raw error the next time a query happens to fail — no new polling cycle was added; detection instead pattern-matches the *shape* of a real query/attach failure against known network-error signatures, exactly as originally proposed.

- **New `src/shared/connection-health.ts`** (no `vscode` import, plain-Node unit-testable) — `isConnectionLostError(error)` recognizes a dropped connection by `.code` (`ECONNRESET`, `EPIPE`, `ECONNREFUSED`, `ECONNABORTED`, `ETIMEDOUT`, `EHOSTUNREACH`, `ENETUNREACH`, `ENOTFOUND`, `ENETDOWN`) when it survives, and falls back to matching those same codes (plus Firebird's own "unable to complete network request", "network error", "socket hang up", etc.) as message substrings — verified by reading the installed `node-firebird` package's own source (`lib/wire/connection.js`'s `_bind_events()`): it re-emits the raw Node.js socket error via `self.db.emit('error', e)`, so `.code` is intact at the `createConnection()`/attach layer, but `NodeClient.queryPromise()`'s own transaction-error path rejects with a **plain string** (`"Error queryPromise: " + err.message`), losing `.code` — Node's default socket-error messages (e.g. `"read ECONNRESET"`) embed the code as text regardless, which is why message-matching is the primary, not just a fallback, path. An ordinary SQL error (syntax, constraint violation) matches neither and is correctly left alone. The same file also holds a tiny in-memory `Set<connectionId>` registry (`markConnectionUnreachable()`/`markConnectionReachable()`/`isConnectionUnreachable()`) — not persisted, resets on reload, same as every other in-memory connection-state tracker in this codebase (e.g. `PooledClient`'s own pooled-socket state).
- **Phase 1 — detection.** `Driver.runQuery()`/`Driver.runBatch()` (`src/shared/driver.ts`) call `Global.reportConnectionOutcome(connectionId, err)` around every `createConnection()`/`queryPromise()` call on the SQL-execution path (the one Firebird Studio already exercises constantly during normal use, per the original proposal) — a lost-connection-shaped error marks that connection id unreachable; any other outcome (success, or a non-connection-shaped error like a syntax error) leaves the registry alone or clears it. `NodeCategoryFolder.getChildren()` (`src/nodes/node-category-folder.ts`) does the same around tree expansion, independently of the SQL-editor path.
- **Phase 2 — status bar + Reconnect.** `Global.updateStatusBarItems()` now checks `isConnectionUnreachable(activeConnection.id)` and, when true, shows `$(debug-disconnect) FIREBIRD: <db> (connection lost)` with `statusBarItem.warningForeground`/`warningBackground` colors instead of its normal text/color, and its click command switches from `firebird.chooseActive` to the new `firebird.reconnectActive` — a real connect-then-detach probe reusing `attemptConnection()` (`src/shared/connection-wizard.ts`, the same one the connection wizard's own "Test Connection" step already uses), reporting the outcome back through the same `reportConnectionOutcome()` path.
- **Phase 3 — tree-node badge.** `NodeDatabase.getTreeItem()` (`src/nodes/node-database.ts`) adds a `"⚠ connection lost"` `description` and swaps its icon to `$(debug-disconnect)` in `errorForeground` when `isConnectionUnreachable(this.dbDetails.id)` — set the moment `NodeCategoryFolder.getChildren()` catches a lost-connection-shaped error while expanding any of that database's category folders, cleared the moment one succeeds again. Both transitions trigger `firebird.explorer.refresh` (the same command `setFilter()`/`clearFilter()` already use) so the badge/icon actually repaints without waiting for an unrelated tree refresh — gated on the registry call actually having changed state, so a normal successful/failed expand that doesn't change anything doesn't cause an extra refresh loop.
- **Central orchestration lives on `Global`** (`Global.reportConnectionOutcome(id, err)`) rather than duplicated at each call site — it already owns the status bar, and now also owns triggering a tree refresh when the shared unreachable-registry state changes; `connection-health.ts` itself stays free of any `vscode` import so its detection logic and registry are unit-testable without a mocked `vscode` module.

### Testing

`src/test/connection-health.test.ts` (14 tests) covers `isConnectionLostError()` (a `.code`-bearing `Error`, a plain string with an embedded code substring — the exact shape `NodeClient.queryPromise()` actually rejects with, `ECONNREFUSED`, Firebird's own "unable to complete network request" wire message, an ordinary SQL syntax/constraint error correctly *not* matching, `undefined`/`null`, and a plain error-shaped object with no `Error` prototype) and the unreachable-connection registry (mark/clear, idempotent no-op reporting on a redundant mark, `undefined` id as a no-op). `Global.reportConnectionOutcome()`/`updateStatusBarItems()`'s warning state and the `firebird.reconnectActive` command are VS Code dialog/status-bar orchestration and aren't independently unit- or suite-tested, matching this codebase's established boundary — pure detection and registry logic are tested directly; the UI wiring built on top was verified by reading through the call sites (`Driver.runQuery()`/`runBatch()`'s `createConnection()`/`queryPromise()` try/catch blocks, `NodeCategoryFolder.getChildren()`'s try/catch, `NodeDatabase.getTreeItem()`'s badge branch) rather than against a live dropped connection, since reliably forcing a real mid-session Firebird connection drop (vs. just a wrong password or unreachable host, both already covered by the existing `attemptConnection()` suite tests) isn't practical to script against the sandbox's real server. ## Proposed feature (original)

- A lightweight periodic health check for the *active* connection only (not every saved connection — pinging every saved connection continuously would be wasteful and is closer to what the Live Profiler's own polling already does for a connection you've deliberately opened that view for) — e.g. a cheap `SELECT 1 FROM RDB$DATABASE` on an interval, or better: react to an actual query failure that looks like a dropped-connection error (`node-firebird`'s connection-reset/ECONNRESET-style errors) rather than adding a new polling cycle at all, since Firebird Studio already runs queries against the active connection constantly during normal use — the same distinction `PooledClient`'s own idle-timeout logic likely already has to make between "genuinely gone" and "just idle."
- Status bar item (`Global.updateStatusBarItems()`) gains a distinct visual state — a warning-colored background/icon (`$(warning)` or `$(debug-disconnect)`) — when the active connection is known to be down, versus its normal state.
- A tree-node-level indicator (a warning badge/`description` suffix on the affected `NodeDatabase`/`NodeHost` item) for a *saved* connection specifically known to be unreachable — likely only meaningfully knowable for whichever connection is currently active, or one the user just tried to expand and got a connection error from (record that outcome against the node rather than a fresh live check).
- A one-click "Reconnect" action from either surface, reusing the same connect path `NodeDatabase.setActive()` already uses.
