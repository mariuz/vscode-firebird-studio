# Visual Schema Designer

**Inspired by**: [vscode-mssql](https://github.com/microsoft/vscode-mssql)'s Schema Designer (visual modeling, drag-and-drop, auto-layout, T-SQL script generation) and its Copilot integration ("GitHub Copilot in Schema Designer").

## Current state in Firebird Studio

**All four phases are done.** `src/schema-designer/` is a single merged webview replacing what used to be two separate, more limited pieces (`src/schema-visualizer/`, a read-only diagram, and `src/table-designer/`, a single-table create/alter form — both deleted):

- `schema-designer/schema-graph.ts` — `buildSchemaGraph()` assembles `getSchemaColumnsQuery()`/`getForeignKeysQuery()` rows into a `SchemaGraph` (tables, columns, relationships), now also carrying each column's default value.
- `schema-designer/index.ts` — `SchemaDesigner` (a `QueryResultsView`) with three entry points, all loading the **full** schema regardless of how they're invoked (this repo's schemas are modest in size, and the visualizer this replaces already always loaded everything in one go): `openFullSchema()` (was `SchemaVisualizer.open()`), `openNewTable()` (was `TableDesigner.open()`), `openForAlterTable(tableName)` (was `TableDesigner.openForAlter()` — `NodeTable#alterTable()` no longer pre-fetches table info itself; the webview does that as part of its normal load). All three back the same three commands as before (`firebird.schemaVisualizer.open`, `firebird.table.createTable`, `firebird.table.alterTable`) — command ids are unchanged, only their handlers now point at one merged webview. Also handles the `askCopilot` message (phase 4, below): calls `vscode.lm.selectChatModels()` directly (this isn't inside a chat-participant callback, so there's no `request.model` handed in — the Language Model API is called the same way any other extension code would) and relays the parsed JSON edit back to the webview.
- `schema-designer/htmlContent/` — the webview: an SVG canvas (pan/zoom/drag/minimap/force-directed auto-layout, forked from the old visualizer) plus an HTML inspector side panel (column editor, forked from the old table designer) for whichever table is selected. Supports adding new tables/columns, altering existing tables' columns and primary key, and drawing/deleting foreign key relationships between columns. **Does not** support dropping an existing table from the canvas — that's a deliberate scope cut; dropping stays `NodeTable.dropTable()`, a separate tree action with its own confirmation dialog, avoiding the DROP-ordering/cascading-FK complexity a canvas-based drop would add.

### Design decisions worth knowing before touching this code

- **`draftGraph` (a plain JS object tree in the webview) is the sole source of truth** for both rendering and DDL generation — the inspector panel is a transient view, synced from/to the selected table's entry on selection change and every field edit. This is a deliberate departure from the single-table designer's original `dataset.original`-on-DOM-rows approach, which only worked there because exactly one table's rows were ever in the DOM at once.
- **Tables carry a stable synthetic `id`** (assigned once, never derived from the name), and **relationships reference columns by direct object identity** (`fromColumn`/`toColumn` point at the actual column object), not by name/id lookup. Both choices exist for the same reason: names are editable (a new table's name always; any table's column names, via rename detection) — a relationship keyed by a name or looked up by name would silently go stale the moment that name changed. This was caught by a design review before implementation and is covered by a regression case in the manual verification harness (`tmp`-scoped, not committed — see Testing below).
- **A table's primary key can't be changed while a live foreign key still references it** — Firebird refuses to `DROP CONSTRAINT`/`ADD PRIMARY KEY` in that situation. The DDL generator detects this (any *kept* relationship whose target table's PK column set is changing) and drops the dependent FK first, re-adding it by its *original* constraint name after the PK change — also caught by review before implementation, also covered in the verification harness.
- **`executeDDL` runs via `Driver.runBatch()`, not `Driver.runQuery()`** — the whole-schema DDL is routinely multi-statement, and `runBatch()` is what actually splits/executes one statement at a time (`runQuery()` doesn't). This was a latent bug in the single-table designer's original alter-mode code (low-exposure there, since single-table alters were often 0-2 statements) that's fixed here rather than carried forward.
- **Reusing `src/schema-diff/schema-diff.ts` for the diff engine was considered and deliberately rejected** — despite what an earlier draft of this doc suggested. It's built for comparing two *live* database connections by table/column name only, with no rename detection and no DDL emission — exactly the drop-and-recreate-on-rename failure mode the `original`-tracking design above exists to avoid. If you're tempted to consolidate these two diffing approaches, that's the reason not to.

- **Copilot schema editing (phase 4)**: an "Ask Copilot" panel in the toolbar sends a plain-text summary of the current draft (built by `serializeSchemaSummary()` — includes tables/columns/relationships added this session but not yet in the database) plus the user's free-text instruction to the model, and asks it to return a small **structured JSON edit** (`{tables: [...], relationships: [...], explanation}`) rather than raw DDL. `applyCopilotEdit()` then applies that JSON as ordinary `draftGraph` mutations — the exact same mutations a user's own inspector edits produce — so the already-proven rename-safe `buildDDL()` diff engine handles turning the result into DDL; the model never has to get Firebird DDL syntax or statement ordering right itself. Deliberately trusts *existence* over the model's self-reported `action` field: whether a table/column is treated as new is decided by whether it's already in the draft, not by whichever `"add"`/`"modify"` label the model attached, so a mislabeled response still produces the correct result.

### Testing

`src/test/schema-graph.test.ts` covers `buildSchemaGraph()`/`normalizeDefault()` (pure, fake rows); `src/test/suite/schema-designer-integration.test.ts` covers the same against a real Firebird server. The webview's inline JS (canvas rendering, inspector, `buildDDL()`, and `applyCopilotEdit()`) previously had **no automated test coverage** at all, matching this repo's general convention for webview inline JS — it was only ever verified with a one-off, uncommitted Node.js harness during development. That gap is now closed for the diff-engine/DDL-generation logic specifically (still the highest-stakes correctness surface in the module — bugs here mean generated DDL that either silently loses data or fails at execution): `src/test/schema-designer-webview.test.ts` (33 tests) drives the same existing `module.exports.__test__` hook under a generic stub DOM (`src/test/webview-harness.ts`, shared with the other three webview app.js files), covering `handleSchemaData()`, and — the scenario set the original uncommitted harness checked — `buildDDL()`'s rename/type/default/not-null/add/drop-column handling, PK changes (including the FK-cycling case from the two review-found scenarios above), add/delete relationships, `serializeSchemaSummary()`, and `applyCopilotEdit()`'s new-table/modify-existing-column/cascading-remove/dangling-reference-ignored cases. `render()`/`measureAll()` (genuine SVG layout/text-measurement) are still not meaningfully testable against a stub DOM and remain unautomated — only the data-level diff engine is covered.
