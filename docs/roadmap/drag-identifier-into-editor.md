# Drag Object Explorer Entity into Editor

**Inspired by**: [vscode-pgsql](https://github.com/microsoft/vscode-pgsql) (1.9.0) — "Dragging an Object Explorer entity into an editor now inserts a correctly double-quoted identifier."

## Current state in Firebird Studio

**Done, both phases.**

- **Phase 1 — drag controller.** `FirebirdTreeDataProvider` (`src/firebirdTreeDataProvider.ts`) now also implements `vscode.TreeDragAndDropController<FirebirdTree>` (`dragMimeTypes: ["text/plain"]`, `dropMimeTypes: []` — drag-out only, no drop target, so `handleDrop()` is never implemented/called). `extension.ts` switched the Firebird Explorer view specifically from `window.registerTreeDataProvider()` to `window.createTreeView(..., { treeDataProvider, dragAndDropController })`, since `dragAndDropController` is only configurable through the latter API — every other view in the extension (bookmarks, query history, tasks) is untouched and still uses `registerTreeDataProvider()`.
- **Which nodes are draggable** is duck-typed rather than hardcoded in the tree provider: a new optional `getDragIdentifier?(): string` on the `FirebirdTree` interface (`src/interfaces/firebird-tree.interface.ts`), implemented by `NodeTable`, `NodeView`, `NodeField`, `NodeProcedure`, `NodeGenerator`, and `NodeDomain` — exactly the six node types the original proposal scoped phase 1 to. `handleDrag()` calls `node.getDragIdentifier?.()` and simply skips any node that doesn't implement it (a host, database, category folder, or any other node type), rather than needing a type-check allowlist to stay in sync as new node types are added.
- **Phase 2 — quoting.** New `src/shared/identifier-quoting.ts`'s `quoteIdentifierIfNeeded(name)` implements Firebird's real identifier rules directly (not by reusing `sanitizeIdentifier()`, which turned out to be the wrong tool — see below): an unquoted identifier must already be all-uppercase, start with a letter, and contain only `[A-Z0-9_$]`, and must not collide with one of the true reserved words in `firebirdReserved` (`src/language-server/firebird-reserved.ts`) — anything else gets wrapped in `"..."`, with any literal `"` in the name doubled (standard SQL identifier-quoting escaping). `handleDrag()` runs every dragged node's real name through this before setting the `text/plain` payload; multiple simultaneously dragged nodes (multi-select) are joined with `, `.

**Why not reuse `sanitizeIdentifier()` directly**, despite the original proposal suggesting it: that function *mutates* an arbitrary CSV header into a brand-new valid identifier (replacing disallowed characters with `_`) for the flat-file-import feature — it has no "should I quote this" mode, and running a real, already-existing object name through it would corrupt names it should otherwise just quote (e.g. a table literally named `Order Items` would come back as `ORDER_ITEMS`, silently referencing the wrong table). `quoteIdentifierIfNeeded()` is a new, small, separate pure function instead — it mirrors the *character-class* rule sanitizeIdentifier() also encodes (`[A-Z0-9_$]`) rather than calling into it, and adds the reserved-word and case checks that a "does this need quoting" decision actually needs and sanitizeIdentifier() has no reason to.

## Proposed feature (original)

- Implement `vscode.TreeDragAndDropController<FirebirdTree>` on `FirebirdTreeDataProvider`, registered via `TreeView.dragAndDropController` (needs switching from `window.registerTreeDataProvider()` to `window.createTreeView()` for the explorer view specifically, since `dragAndDropController` is only configurable through the latter) — `handleDrag()` sets a MIME payload (`text/plain` is enough; VS Code's editor drop handler already accepts a plain-text drop) to the node's real object name.
- **Correct quoting is the actual point, not just insertion** — Firebird identifiers only need double-quoting when they're not all-uppercase-and-simple (mixed/lower case, containing characters outside `[A-Z0-9_$]`, or colliding with a reserved word) per Firebird's own case-folding rules; an always-quote-everything approach would be technically safe but noisier than necessary for the common all-uppercase-unquoted-identifier case this codebase already assumes elsewhere (e.g. `sanitizeIdentifier()` in `src/shared/flat-file-parser.ts` already encodes similar Firebird identifier rules for a different feature — reuse or mirror that logic rather than re-deriving quoting rules a third time).
- Scope to `NodeTable`/`NodeView`/`NodeProcedure`/`NodeField` (and similar) nodes specifically — dragging a category folder (`NodeCategoryFolder`) or a host/database node has no sensible single-identifier text to insert.

### Testing

`src/test/identifier-quoting.test.ts` (9 tests) covers `quoteIdentifierIfNeeded()`: an all-uppercase simple name and one with digits/underscore/`$` both pass through unquoted; lowercase, mixed-case, and a name containing a space are quoted; a name starting with a digit is quoted; a name colliding with a reserved word (`ORDER`, `SELECT`, `GROUP`) is quoted even though its shape alone would otherwise be valid unquoted; a name that's merely a reserved-word *substring* (`ORDERS`, not `ORDER`) is correctly left unquoted; and a literal `"` in a name is escaped by doubling it. `FirebirdTreeDataProvider`'s `handleDrag()`, the `window.createTreeView()` switch, and each node class's new `getDragIdentifier()` are VS Code tree/drag-and-drop orchestration and aren't independently unit- or suite-tested (there's no `TreeDragAndDropController` test harness available, mocked or real) — matching this codebase's established boundary between pure/tested logic and orchestration verified by reading the call sites. The `window.createTreeView()` swap itself *is* indirectly covered by the existing suite: the real Extension Development Host suite's activation test exercises real view registration end-to-end, and continued passing after the swap confirms it didn't break view registration.
