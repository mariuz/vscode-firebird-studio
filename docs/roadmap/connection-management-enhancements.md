# Connection Dialog & Management Enhancements

**Inspired by**: [vscode-mssql](https://github.com/microsoft/vscode-mssql)'s Connection Dialog work — testing a connection before saving it and distinguishing "editing an existing connection" from "creating a new one" (1.42.0), copying a saved connection profile's connection string directly from Object Explorer (1.38.0), and a dedicated background task panel showing active/completed long-running operations with real-time progress (1.42.0, originally for container/Fabric provisioning).

## Current state in Firebird Studio

**Phases 1 through 3 are done.** The relevant existing pieces:

- `src/shared/connection-wizard.ts` — a `MultiStepInput`-driven wizard (`collectInputs()` → `connectionType`/`dockerContainer`/`host`/`database`/`port`/`user`/`password`/`role`/`wireCrypt`/`sshTunnel` steps) that collects every field and hands back a `ConnectionOptions`.
- `NodeDatabase` (`src/nodes/node-database.ts`) already has a rich set of per-connection commands (`setPassword`, `setConnectionGroup`, `renameDatabase`, `dropDatabase`, `backupDatabase`/`restoreDatabase`).
- `src/container-provisioning/index.ts` reports progress via a single `window.withProgress({location: ProgressLocation.Notification, ...})` call — a transient toast that disappears once the container is up (or the wizard is dismissed), with no way to check back on it, and no shared home for a second concurrent long operation (e.g. a backup running at the same time) to report into.
- **Phase 1 — Test Connection step — done.** A new terminal step in the wizard, reached after every field is collected: for embedded connections (after `role()`) and for non-SSH network/Docker connections (after choosing "No" on the SSH tunnel step). Offers "$(plug) Test Connection" or "$(check) Save Without Testing" (the default on Escape/dismiss, so nothing changes for anyone who ignores the new step); a real `attemptConnection()` connect-then-detach runs inside a `window.withProgress` notification, reporting success via `showInformationMessage` or failure via `showErrorMessage` with "Retry"/"Save Anyway" — never a hard block on saving. **Deliberately skipped for SSH-tunneled connections**: `SshTunnelClient`'s connect path resolves the SSH credential via `CredentialStore.getSshPassword(connectionOptions.id)`, keyed by a *saved* connection's id — which doesn't exist yet at this point in the wizard, so there's no reliable way to test through the tunnel before the connection (and its credential) are actually saved. The SSH-configured branch of the wizard is unchanged, finishing the same way it always did.
- **Phase 2 — Copy Connection String command — done, and deliberately not the firebird:// URL scheme.** New `firebird.database.copyConnectionString` command (right-click menu, same registration pattern as `setPassword`; a new `manage@8` menu entry) → `NodeDatabase.copyConnectionString()` → new `buildConnectionString()` in `src/shared/connection-string.ts` (alongside `parseConnectionString()`) → `env.clipboard.writeText()`. Uses Firebird's own native DSN shape (`host/port:database`, or a bare `database` path for embedded) rather than this extension's own `firebird://` URL scheme — that scheme requires a hostname (`parseConnectionString()` rejects anything without one) and so has no representation for an embedded connection at all, which this needs to handle too; the native DSN shape naturally covers both and is what `isql -c`/JDBC/ODBC connection strings already use, so it's immediately recognizable to any Firebird user. Never includes the password, matching this repo's "password never leaves SecretStorage casually" posture; the non-secret username, when set, is called out on its own line, with an explicit note that the password was left out on purpose.
- **Phase 3 — Distinguish edit vs. create — done.** New `firebird.database.editConnection` command ("Edit Connection..." on a database's right-click menu, `second@2`) → `NodeDatabase.editConnection()`, which resolves the real password (`getResolvedConnectionDetails()`, needed so the wizard's password field prefills correctly and an in-wizard Test Connection works without retyping an unchanged password) and calls `connectionWizard()` with a new optional `existing: ConnectionOptions` second parameter. When set: the "paste a connection string" prompt and the connection-type/Docker-discovery step are both skipped (jumping straight to `database()`/`host()` depending on `existing.embedded` — changing a connection's *type* isn't supported here, only its fields), and every text field (`host`/`database`/`port`/`user`/`password`/`role`) pre-fills via `showInputBox`'s existing `value` option, so leaving a step unchanged keeps the current value instead of resetting to the wizard's create-mode defaults (e.g. blank password no longer means "masterkey" when editing — it means "whatever's already there, untouched"). On completion, the result is saved back over the *same* connection id (not a new `uuidv1()` one) — the stored password is re-stored (a no-op if unchanged), the SSH tunnel credential is updated or deleted depending on whether `sshTunnel` is still set, and `Global.activeConnection` is refreshed only if the *edited* connection was already the active one (editing an inactive connection no longer surprise-switches which one is active). **Two small correctness fixes needed along the way**, both specific to edit mode reusing the wizard's existing steps: `wireCrypt()` previously only ever *set* a non-default value, never explicitly cleared one — editing a `Disabled` connection back to `Enabled` wouldn't have taken effect without this; `sshTunnel()`'s "No" branch previously never cleared a pre-populated `sshTunnel`/`sshTunnelPassword` — removing SSH tunneling from an existing connection wouldn't have worked without this. Both now distinguish an explicit choice (clear) from dismissing the step (Escape — leave the pre-populated value alone, edit mode's "unchanged" default). **Deliberately out of scope**: reusing a Docker-discovered container's connection type, and refused entirely for workspace-sourced (`.vscode/firebird.json`) connections, which aren't in `globalState` at all — same guard message `renameDatabase()` already uses ("edit or remove it there instead").
- Not done yet: the background task panel.

### Testing

The wizard itself remains untested (VS Code dialog orchestration — same boundary this repo already draws for `node-database.ts`'s wizard-style methods, per `flat-file-import-wizard.md`'s precedent); none of the three phases change that, including phase 3's own new interactive pre-fill logic (`wireCrypt()`'s/`sshTunnel()`'s fixes included) — genuinely untestable without either simulating keystrokes through real `QuickPick`/`InputBox` widgets or refactoring the wizard's save logic out from its interactive collection logic purely to make it testable, which this repo's stated preference (no abstractions beyond what's needed) argues against doing just for coverage's sake. `attemptConnection()` (phase 1) is exported and covered by a new `src/test/suite/connection-wizard-integration.test.ts`: a real successful connection to the seeded test server, a wrong password, and an unreachable host — all three confirmed to resolve to a plain string (or `undefined`) rather than rejecting, against a real Firebird server. `buildConnectionString()` (phase 2) is a pure function and gained 7 unit tests in `src/test/connection-string.test.ts`: the network DSN shape, the port-omitted case, the embedded case, that the password genuinely never appears in the output, the user-line's presence/absence, and the "password not included" note always being there. Both `firebird.database.copyConnectionString` and `firebird.database.editConnection`'s registration are covered by `src/test/suite/extension.test.ts`'s existing "every roadmap doc's key command(s) are actually registered" suite (the `connection-management-enhancements.md` entry).

## Proposed feature

1. ~~**Test Connection step.**~~ — **done**, see above (implemented as the wizard's own new terminal step rather than a separate follow-up dialog, so it's reached naturally at the point every field is already known).
2. ~~**Distinguish edit vs. create.**~~ — **done**, see above (implemented as an optional `existing` parameter to `connectionWizard()` rather than a `mode: 'create' | 'edit'` string as originally sketched here — the presence/absence of `existing` already unambiguously signals which mode, so a separate mode flag would have been redundant).
3. ~~**Copy Connection String command.**~~ — **done**, see above (built on Firebird's native DSN shape rather than the `firebird://` scheme originally guessed here — see above for why).
4. **Background task panel.** A `TreeDataProvider`-backed view (or a `WebviewView` in the same activity-bar container as the Firebird explorer) listing active and recently-completed long-running operations — container provisioning, backup/restore, database project publish — each reporting into it instead of (or alongside) today's one-off `withProgress` notifications, so a user who dismisses or misses the toast can still check whether a backup actually finished. Lowest priority of the four here: the existing per-operation notifications already work, this is purely a discoverability improvement once more than one long-running feature exists side by side.
