# Firebird Database Projects (schema-as-code)

**Inspired by**: [vscode-mssql](https://github.com/microsoft/vscode-mssql)'s SQL Database Projects extension (offline `.sqlproj` authoring, build, and publish) and DACPAC/BACPAC deploy/extract.

## Current state in Firebird Studio

**All four phases are done.** Right-click a database → **Extract Database Project...** writes the connected schema out as one `.sql` file per object; **Build Database Project...** (Command Palette — it operates on a folder, not a live connection) concatenates an extracted project into one reviewable deploy script; **Publish Database Project...** (Command Palette) diffs a saved project against a live target connection and generates an executable migration script. Extract/Build/Publish now cover domains, roles, exceptions, and users alongside tables/views/procedures/triggers/generators (phase 4, below).

- `src/database-projects/project-model.ts` — pure DDL-reconstruction/manifest logic (no vscode/Driver dependency, unit-tested like `schema-graph.ts`/`openapi-spec.ts`): `buildTableCreateDDL()` reconstructs a `CREATE TABLE` from the same `SchemaColumn[]` model the Schema Designer already assembles (columns, `NOT NULL`, `DEFAULT`, a trailing `PRIMARY KEY (...)` clause); `buildForeignKeyDDL()` builds an `ALTER TABLE ... ADD CONSTRAINT ... FOREIGN KEY` per relationship; `buildProcedureCreateDDL()`/`buildTriggerCreateDDL()`/`buildViewCreateDDL()` wrap each object's already-complete `RDB$PROCEDURE_SOURCE`/`RDB$TRIGGER_SOURCE`/`RDB$VIEW_SOURCE` text in a `CREATE OR ALTER` header; `buildDomainCreateDDL()`/`buildExceptionCreateDDL()`/`buildRoleCreateDDL()`/`buildUserCreateDDL()` do the same for the phase-4 object types (see below); `buildProjectFiles()` assembles everything (generators and domains first, then tables, then `foreign-keys.sql`, then exceptions, then views/procedures/triggers, then roles/users) plus a `firebird.project.json` manifest recording that same dependency-safe order, which the Build step just replays.
- `src/database-projects/index.ts` — `fetchProjectSnapshot(connectionOptions)` fetches a live connection's full schema (tables/columns/FKs/domains/views/procedures/triggers/exceptions/generators/roles/users/PK constraint names) into the structured `ProjectInput` shape, shared by both Extract and Publish. `runExtractProject()` calls it and writes every project file *plus* a raw `firebird.project-snapshot.json` (the exact `ProjectInput`, serialized) so Publish can later diff this precise point-in-time snapshot without re-parsing the generated `.sql` files or needing the source connection to still be reachable. `runBuildProject()` reads an existing project's manifest, concatenates its files in the recorded order, and opens the result for review. `runPublishProject()` (Phase 3) reads a project's snapshot, prompts for a target connection, fetches the target's *live* schema into the same shape, diffs the two, and opens an executable migration script for review — none of these ever execute anything automatically, matching this extension's "always show generated DDL before running it" convention.
- `src/database-projects/publish-model.ts` — pure diff (`diffProjects()`) and script generation (`buildPublishScript()`), unit-tested plus full real-Firebird-server round-trip tests (`src/test/suite/database-project-publish-integration.test.ts`, two of them — one for tables/views/procedures/triggers/generators/FKs, one for domains/exceptions/roles/users) that each generate a migration script and *actually execute it*, then re-fetch the target and confirm it now matches the source (including a parameterized procedure and a procedure that genuinely raises a restored exception — see below). Diffing is **name-based only** (tables/columns/procedures/etc. matched by name, not by any rename-tracking) — a renamed column/table shows up as a drop of the old name plus an add of the new one, the same limitation `schema-diff.ts`'s existing connection-vs-connection comparison already has, not a heuristic worth inventing.
- **Parameterized procedures reconstruct correctly.** `getAllProcedureParametersQuery()`/`procedureParametersQuery()` fetch `RDB$PROCEDURE_PARAMETERS` (joined to `RDB$FIELDS` for type/length/NUMERIC-DECIMAL precision, the same join `tableInfoQuery()` uses), and `ProcedureSource.parameters` carries them through `ProjectInput`. `buildProcedureCreateDDL()`/`buildProcedureParameterHeader()` reconstruct the `(in params...)`/`RETURNS (out params...)` header — `RDB$PROCEDURE_SOURCE` itself never includes either (confirmed live), so there was no other way to get this right. `NodeProcedure#editProcedure()`'s `ALTER PROCEDURE` scaffold needed the same fix for a different reason: unlike `ALTER TRIGGER` (body-only changes are fine, confirmed live), a plain `ALTER PROCEDURE` genuinely requires re-specifying the *entire* parameter list even to just change the body — omitting it makes every parameter "unknown" inside the new body.

### Four real bugs found and fixed while building and *actually executing* Publish's generated scripts (not just reviewing them)

Extract/Build's output had only ever been manually reviewed before Publish came along and made it necessary to actually run a generated script end-to-end — surfacing four genuine defects, three of them pre-existing in the already-shipped Extract/Build feature:

1. **`DROP COLUMN` isn't valid Firebird syntax** — the correct form is `ALTER TABLE t DROP <column>`, no `COLUMN` keyword (confirmed directly against a live server). `buildPublishScript()`'s column-drop statement was wrong; fixed.
2. **`src/shared/sql-splitter.ts` mis-split a PSQL block preceded by a comment.** Both its `SET TERM` and bare-PSQL-header detection only special-cased a genuine statement boundary via `current.trim() === ""` — which is false the moment a leading `-- comment` (with no `SET TERM`) has already been accumulated, silently disabling the BEGIN/END depth tracking that keeps a procedure/trigger body's internal semicolons from being mistaken for statement terminators. Fixed by stripping comments before the blank check (`isBlankOrCommentsOnly()`) — a real fix to shared, heavily-used infrastructure, not something scoped to Publish alone.
3. **`RDB$PROCEDURE_SOURCE` never includes the `AS` keyword, and `RDB$TRIGGER_SOURCE` never includes the `FOR <table> {ACTIVE|INACTIVE} {BEFORE|AFTER} <event>` header** — both confirmed directly against a live server. A prior version of this doc/code claimed otherwise ("these source columns already contain everything after the object's name ... AS ... confirmed against `NodeProcedure#editProcedure()`") — that claim was never actually exercised by execution, only by a human reviewing an opened scaffold, so it went unnoticed. **This means every trigger Extract/Build ever reconstructed was missing its required header — a syntax error the instant anyone tried to actually run it**, not a Publish-specific bug. Fixed in `buildProcedureCreateDDL()`/`buildTriggerCreateDDL()` (a new `describeTriggerType()` decodes `RDB$TRIGGER_TYPE`, replacing a private duplicate in `NodeTrigger`) and in `NodeProcedure#editProcedure()`'s scaffold.
4. **A parameterized procedure's parameter list/`RETURNS` clause was silently dropped entirely** — same root cause as #3 (`RDB$PROCEDURE_SOURCE` excludes it), fixed the same day once found rather than left as a follow-up, since the query/data-model work was already fresh. See above.

### Phase 4 — Domains, roles, exceptions, and users

Extract/Build/Publish now cover `DomainSource`/`RoleSource`/`ExceptionSource`/`UserSource` alongside the existing object types, added to `ProjectInput` and threaded through `fetchProjectSnapshot()`, `buildProjectFiles()`, `diffProjects()`, and `buildPublishScript()`.

- **Domains**: `getDomainsQuery()` now also selects `RDB$DEFAULT_SOURCE`/`RDB$VALIDATION_SOURCE` (cast the same way procedure/trigger/view source text already is elsewhere in `queries.ts`) — confirmed directly against a live server that `RDB$VALIDATION_SOURCE` already comes back as complete `CHECK (...)` text, ready to append as-is, while `RDB$DEFAULT_SOURCE` needs its `"DEFAULT "` prefix stripped the same way `schema-graph.ts`'s `normalizeDefault()` already does for table columns (reused directly). `buildDomainCreateDDL()` builds a plain `CREATE DOMAIN` for a new one; a *changed* domain instead gets one `ALTER DOMAIN` statement per changed aspect (`buildDomainAlterStatements()` in `publish-model.ts`) — confirmed live that Firebird has **no** `CREATE OR ALTER DOMAIN` (only PROCEDURE/TRIGGER/VIEW/EXCEPTION get that shorthand) and that a domain allows at most one `CHECK` constraint at a time, so a changed check is `DROP CONSTRAINT` (only if one existed) then `ADD CONSTRAINT` (only if the new one exists), not a single combined statement.
- **Exceptions**: straightforward — `CREATE OR ALTER EXCEPTION name 'message'` (confirmed live that, unlike DOMAIN/ROLE, EXCEPTION *does* support `CREATE OR ALTER`), with the message string escaped via `escapeSqlLiteral()` (exported from `queries.ts`, previously private there — reused rather than a second copy).
- **Roles**: existence only (`CREATE ROLE name`) — Firebird roles have no other extractable properties, and GRANT/membership migration is a distinct, deliberately out-of-scope concern (see below).
- **Users**: name only (`SEC$USERS`, via the pre-existing `getUsersQuery()`) — **and confirmed live that Firebird users/logins are server-wide** (the single security database shared by every attachment on that server instance), unlike roles, which are per-database (`RDB$ROLES`). `buildUserCreateDDL()` deliberately emits a **commented-out** `CREATE USER name PASSWORD 'ChangeMe123!';` with a `TODO` — Firebird's security database stores password hashes, never a recoverable plaintext value, so there's no way to extract or preserve a real password, and emitting an *active* statement would mean silently creating a real, log-in-capable account with a publicly-known placeholder password the instant someone ran the generated script without reading every line — the same kind of silent, security-relevant side effect this extension's "always show generated DDL for review" convention exists to prevent. A commented line stays inert even if the rest of the script is run wholesale, while still documenting exactly which user existed and the statement to uncomment (with a real password) to recreate it.
- **Ordering**: `buildProjectFiles()`/`buildPublishScript()` place exceptions before views/procedures/triggers — confirmed directly against a live server that `CREATE PROCEDURE`/`CREATE TRIGGER` fails outright ("exception ... not defined") if a body's `EXCEPTION name;` statement names one that doesn't exist yet, and the reverse holds for drops (procedures/triggers must be dropped before an exception they reference). Domains and generators are placed even earlier, ahead of tables — domains as the conventional "define your types before your tables" order (though `getSchemaColumnsQuery()`'s columns already resolve to a domain's flattened base type rather than referencing it by name, so this isn't a hard dependency for what's actually generated today), and **generators for a real reason**: see the bugs below.

#### Three more real bugs found and fixed, this time while live-verifying phase 4 specifically (Extract → Build → actually execute → re-diff → Publish against injected drift → actually execute)

1. **Generators were emitted dead last** (after procedures/triggers), a **pre-existing** bug that predates this phase entirely — confirmed live that running a generated script fails with `"Generator ... is not defined"` the moment a trigger body or column `DEFAULT` calls `GEN_ID()` on a generator whose own `CREATE SEQUENCE` hasn't run yet (a common auto-incrementing-key pattern). Fixed by moving generators to the front, in both `buildProjectFiles()` and `buildPublishScript()`.
2. **`RDB$FIELD_LENGTH` is a column's storage size in *bytes*, not characters** — a genuinely significant, **pre-existing** gap reaching well beyond this phase. Confirmed directly against a live Firebird 6.0 server: a UTF8-charset `VARCHAR(150)` reports `RDB$FIELD_LENGTH = 600` (150 × 4 bytes/char); `RDB$CHARACTER_LENGTH` correctly reports `150`, and is non-null *only* for `CHAR`/`VARCHAR` (confirmed live across every other type — `INTEGER`/`BIGINT`/`BLOB`/`NUMERIC`/`TIMESTAMP`/`BOOLEAN` all have it `NULL`) — exactly the two types every consumer of `FIELD_LENGTH` actually uses it for. This was caught by Publish's own round-trip test re-diffing a database it had just rebuilt from its own Extract output and finding a spurious "changed" domain/column purely from the 4×-inflated length (each round trip would have *compounded* the inflation further, since `CREATE DOMAIN x AS VARCHAR(600)` genuinely declares 600 *characters*, not 600 bytes). Fixed with `COALESCE(RDB$CHARACTER_LENGTH, RDB$FIELD_LENGTH)` everywhere `RDB$FIELD_LENGTH AS FIELD_LENGTH` appeared in `queries.ts` — six other query functions beyond `getDomainsQuery()` share this exact pattern (`tableInfoQuery()`, `fieldsQuery()`, `viewColumnsQuery()`, `procedureParametersQuery()`, `getAllProcedureParametersQuery()`, `getSchemaColumnsQuery()`), all fixed identically since the COALESCE is safe everywhere (verified live, per above) and every one of them was equally wrong for a UTF8-charset database — including `tableInfoQuery()`, which backs the tree explorer's own field-length display, and `getSchemaColumnsQuery()`, used well beyond Database Projects (Schema Designer, Data API Builder's OpenAPI `maxLength`, schema-diff).
3. **A comment-only "statement" isn't valid SQL to send to the server.** `buildUserCreateDDL()`'s deliberately-commented-out `CREATE USER` (see above) means a Build/Publish script can contain a chunk that, once line/block comments are stripped, has nothing left — confirmed live that Firebird rejects this with `"Dynamic SQL Error ... Unexpected end of command"` rather than accepting it as a no-op. `src/shared/sql-splitter.ts#splitStatements()`'s `flush()` already filtered *blank* chunks but not *comment-only* ones (the raw, un-stripped text is non-empty even when it's 100% comment) — fixed by reusing the module's own `isBlankOrCommentsOnly()` helper (already used internally for boundary detection) to filter the final output too. This is shared, general-purpose infrastructure, not scoped to Database Projects alone — e.g. `validateReadOnlyStatement()` (the MCP server's read-only guard) now correctly reports `"No SQL statement found"` for a comment-only input instead of the previously-misleading `"Only SELECT..."` (which implied a statement of the wrong kind was found, when really none was).

None of these three were introduced by writing new, previously-nonexistent code for this phase in the way the original four bugs were (bugs #1 and #2 above are outright pre-existing, predating this phase; #3 is a direct, first-order consequence of this phase's own new `buildUserCreateDDL()` design interacting with pre-existing infrastructure) — all three were caught the same way the original four were: by actually running a generated script against a real server rather than only reviewing it or unit-testing the generator in isolation.

**Verification**: extensive new unit tests for every new pure function (`project-model.test.ts`, `publish-model.test.ts`, plus `sql-splitter.test.ts`/`sql-analysis.test.ts` for the comment-filtering fix), a full local live round-trip (Extract → Build → actually execute against a fresh target → re-fetch and diff against the source, expecting zero drift → inject real drift into the target → Publish → actually execute the generated fix → re-verify) against a real Firebird 6.0 server, and a second `src/test/suite/database-project-publish-integration.test.ts` test (alongside the pre-existing one) running the same kind of round-trip inside a real Extension Development Host (`npm run test:vscode-host`) — including actually invoking a procedure that raises a restored exception and confirming the *exact* message comes back.

A related, **out-of-scope** discovery from getting the Extension Development Host suite to actually run end-to-end for the first time (previously always blocked by a `WireCrypt` mismatch against this environment's local server, before this session): `tableInfoQuery()`'s `MIN(rc.RDB$CONSTRAINT_TYPE)` aggregate (used by the tree explorer's own field listing, not by Database Projects) fails specifically through `node-firebird`'s wire-protocol client (`"string right truncation, expected length 2, actual 11"`) while the *identical* SQL succeeds via `isql`/the native client — a pre-existing `node-firebird` compatibility quirk, unrelated to this phase's work (confirmed reproducible with the query text from *before* the `COALESCE` fix above, and confirmed absent when using the native driver instead), left unfixed as a separate, disclosed follow-up rather than pulled into this phase's scope.

### Explicitly deferred (not done)

- **Column-type-change data safety**: a column type change is emitted as a plain `ALTER COLUMN ... TYPE`, which Firebird accepts for a same-family widening (e.g. `VARCHAR(10)` → `VARCHAR(30)`, confirmed live) but will reject outright for a genuinely incompatible conversion — there is no temp-column-and-copy dance for cases that would need one. This is an acceptable, disclosed gap given the "always review before running" convention: an invalid `ALTER` fails loudly with Firebird's own error the moment it's run, rather than silently corrupting data.
- **Combined multi-event triggers** (Firebird's bitmask encoding for a single trigger firing on more than one event) aren't decoded — `describeTriggerType()` only covers the six simple single-event `RDB$TRIGGER_TYPE` values, matching the scope the pre-existing tree tooltip already had.
- **Workspace-level project binding**: no `.vscode/firebird.json`-style "this workspace's project folder + connection" config; Extract/Build/Publish always prompt for a folder/connection.
- **Role/user GRANT and membership migration**: a role's own privileges (what it's been `GRANT`ed) and which users/roles have been granted a given role aren't extracted or migrated — only the objects' bare existence. A real, separate scope, not attempted here.
- **A `node-firebird` wire-protocol compatibility quirk in `tableInfoQuery()`** (unrelated to this phase, see above) — affects the tree explorer's field listing under the pure-JS driver in at least this environment; not investigated further here.

## Technical notes

- The riskiest new logic — generating *safe, executable* ALTER statements from a diff — turned out to be less about the ALTER statements themselves (Firebird's own ALTER COLUMN/DROP CONSTRAINT/ADD PRIMARY KEY syntax is fairly permissive once you know the exact grammar, verified live rather than assumed) and much more about correctly reconstructing the *unchanged* surrounding DDL (the AS keyword, the trigger event header, statement-splitting past a leading comment) that Extract/Build had silently gotten wrong from the start. Actually executing a generated script — not just eyeballing it — is what surfaced all three.
- FK/PK interaction: a table's primary key can change while a *kept* foreign key still references a *different* column that has its own separate `UNIQUE` constraint, with no need to drop/recreate that FK at all (verified live) — Firebird only requires dropping and re-adding an FK when the PK change actually removes the unique/primary constraint the FK's own referenced column depends on. `buildPublishScript()`'s FK-cycling logic conservatively cycles any kept FK referencing a table whose PK changed, which is always safe (if occasionally more churn than strictly necessary) and was validated at the unit-test level (constraint-name-preserving drop-then-readd, correct statement ordering) rather than against a live server, since constructing a schema-valid live scenario that specifically *requires* cycling (as opposed to a table redesign that's invalid regardless of any tool) turned out to be a genuine edge case once Firebird's actual FK-validity rules were checked directly.
