# Firebird Database Projects (schema-as-code)

**Inspired by**: [vscode-mssql](https://github.com/microsoft/vscode-mssql)'s SQL Database Projects extension (offline `.sqlproj` authoring, build, and publish) and DACPAC/BACPAC deploy/extract.

## Current state in Firebird Studio

**All three phases are done.** Right-click a database → **Extract Database Project...** writes the connected schema out as one `.sql` file per object; **Build Database Project...** (Command Palette — it operates on a folder, not a live connection) concatenates an extracted project into one reviewable deploy script; **Publish Database Project...** (Command Palette) diffs a saved project against a live target connection and generates an executable migration script.

- `src/database-projects/project-model.ts` — pure DDL-reconstruction/manifest logic (no vscode/Driver dependency, unit-tested like `schema-graph.ts`/`openapi-spec.ts`): `buildTableCreateDDL()` reconstructs a `CREATE TABLE` from the same `SchemaColumn[]` model the Schema Designer already assembles (columns, `NOT NULL`, `DEFAULT`, a trailing `PRIMARY KEY (...)` clause); `buildForeignKeyDDL()` builds an `ALTER TABLE ... ADD CONSTRAINT ... FOREIGN KEY` per relationship; `buildProcedureCreateDDL()`/`buildTriggerCreateDDL()`/`buildViewCreateDDL()` wrap each object's already-complete `RDB$PROCEDURE_SOURCE`/`RDB$TRIGGER_SOURCE`/`RDB$VIEW_SOURCE` text in a `CREATE OR ALTER` header; `buildProjectFiles()` assembles everything (tables, then `foreign-keys.sql`, then views/procedures/triggers/generators) plus a `firebird.project.json` manifest recording that same dependency-safe order, which the Build step just replays.
- `src/database-projects/index.ts` — `fetchProjectSnapshot(connectionOptions)` fetches a live connection's full schema (tables/columns/FKs/views/procedures/triggers/generators/PK constraint names) into the structured `ProjectInput` shape, shared by both Extract and Publish. `runExtractProject()` calls it and writes every project file *plus* a raw `firebird.project-snapshot.json` (the exact `ProjectInput`, serialized) so Publish can later diff this precise point-in-time snapshot without re-parsing the generated `.sql` files or needing the source connection to still be reachable. `runBuildProject()` reads an existing project's manifest, concatenates its files in the recorded order, and opens the result for review. `runPublishProject()` (Phase 3) reads a project's snapshot, prompts for a target connection, fetches the target's *live* schema into the same shape, diffs the two, and opens an executable migration script for review — none of these ever execute anything automatically, matching this extension's "always show generated DDL before running it" convention.
- `src/database-projects/publish-model.ts` — pure diff (`diffProjects()`) and script generation (`buildPublishScript()`), unit-tested (28 tests) plus a full real-Firebird-server round-trip test (`src/test/suite/database-project-publish-integration.test.ts`) that generates a migration script and *actually executes it*, then re-fetches the target and confirms it now matches the source. Diffing is **name-based only** (tables/columns/procedures/etc. matched by name, not by any rename-tracking) — a renamed column/table shows up as a drop of the old name plus an add of the new one, the same limitation `schema-diff.ts`'s existing connection-vs-connection comparison already has, not a heuristic worth inventing.
- Domains, roles, exceptions, and users are **not** extracted or migrated (see "Explicitly deferred" below) — one-at-a-time DDL for these object types exists instead via "Generic Script as Create/Drop".

### Three real bugs found and fixed while building and *actually executing* Publish's generated scripts (not just reviewing them)

Extract/Build's output had only ever been manually reviewed before Publish came along and made it necessary to actually run a generated script end-to-end — surfacing three genuine defects, two of them pre-existing in the already-shipped Extract/Build feature:

1. **`DROP COLUMN` isn't valid Firebird syntax** — the correct form is `ALTER TABLE t DROP <column>`, no `COLUMN` keyword (confirmed directly against a live server). `buildPublishScript()`'s column-drop statement was wrong; fixed.
2. **`src/shared/sql-splitter.ts` mis-split a PSQL block preceded by a comment.** Both its `SET TERM` and bare-PSQL-header detection only special-cased a genuine statement boundary via `current.trim() === ""` — which is false the moment a leading `-- comment` (with no `SET TERM`) has already been accumulated, silently disabling the BEGIN/END depth tracking that keeps a procedure/trigger body's internal semicolons from being mistaken for statement terminators. Fixed by stripping comments before the blank check (`isBlankOrCommentsOnly()`) — a real fix to shared, heavily-used infrastructure, not something scoped to Publish alone.
3. **`RDB$PROCEDURE_SOURCE` never includes the `AS` keyword, and `RDB$TRIGGER_SOURCE` never includes the `FOR <table> {ACTIVE|INACTIVE} {BEFORE|AFTER} <event>` header** — both confirmed directly against a live server. A prior version of this doc/code claimed otherwise ("these source columns already contain everything after the object's name ... AS ... confirmed against `NodeProcedure#editProcedure()`") — that claim was never actually exercised by execution, only by a human reviewing an opened scaffold, so it went unnoticed. **This means every trigger Extract/Build ever reconstructed was missing its required header — a syntax error the instant anyone tried to actually run it**, not a Publish-specific bug. Fixed in `buildProcedureCreateDDL()`/`buildTriggerCreateDDL()` (a new `describeTriggerType()` decodes `RDB$TRIGGER_TYPE`, replacing a private duplicate in `NodeTrigger`) and in `NodeProcedure#editProcedure()`'s scaffold.

### Explicitly deferred (not done)

- **Parameterized/RETURNS procedure reconstruction**: `RDB$PROCEDURE_SOURCE` excludes a procedure's parameter list and `RETURNS (...)` clause entirely (confirmed directly against a live server across three shapes — none of them include it), and `ProcedureSource`/the underlying query capture no such data at all today. A parameterless procedure (the common case for simple triggers-as-procedures, validation logic, etc.) round-trips correctly through Extract/Build/Publish; **one with input or output parameters does not** — it would generate a syntactically invalid `CREATE OR ALTER PROCEDURE` missing its parameter list, currently undetected (there's no validation catching this before the script is opened for review). Fixing this properly needs a new query against `RDB$PROCEDURE_PARAMETERS` (name/type/direction/position) and parameter-type reconstruction logic mirroring `columnTypeToDDL()` — a real, separate, well-scoped follow-up, deliberately not folded into this pass given how much this one already grew from its original "biggest net-new logic" risk callout.
- **Column-type-change data safety**: a column type change is emitted as a plain `ALTER COLUMN ... TYPE`, which Firebird accepts for a same-family widening (e.g. `VARCHAR(10)` → `VARCHAR(30)`, confirmed live) but will reject outright for a genuinely incompatible conversion — there is no temp-column-and-copy dance for cases that would need one. This is an acceptable, disclosed gap given the "always review before running" convention: an invalid `ALTER` fails loudly with Firebird's own error the moment it's run, rather than silently corrupting data.
- **Workspace-level project binding**: no `.vscode/firebird.json`-style "this workspace's project folder + connection" config; Extract/Build/Publish always prompt for a folder/connection.
- Domains, roles, exceptions, users — one-at-a-time DDL for these object types exists via "Generic Script as Create/Drop" instead of whole-database extraction/migration.

## Technical notes

- The riskiest new logic — generating *safe, executable* ALTER statements from a diff — turned out to be less about the ALTER statements themselves (Firebird's own ALTER COLUMN/DROP CONSTRAINT/ADD PRIMARY KEY syntax is fairly permissive once you know the exact grammar, verified live rather than assumed) and much more about correctly reconstructing the *unchanged* surrounding DDL (the AS keyword, the trigger event header, statement-splitting past a leading comment) that Extract/Build had silently gotten wrong from the start. Actually executing a generated script — not just eyeballing it — is what surfaced all three.
- FK/PK interaction: a table's primary key can change while a *kept* foreign key still references a *different* column that has its own separate `UNIQUE` constraint, with no need to drop/recreate that FK at all (verified live) — Firebird only requires dropping and re-adding an FK when the PK change actually removes the unique/primary constraint the FK's own referenced column depends on. `buildPublishScript()`'s FK-cycling logic conservatively cycles any kept FK referencing a table whose PK changed, which is always safe (if occasionally more churn than strictly necessary) and was validated at the unit-test level (constraint-name-preserving drop-then-readd, correct statement ordering) rather than against a live server, since constructing a schema-valid live scenario that specifically *requires* cycling (as opposed to a table redesign that's invalid regardless of any tool) turned out to be a genuine edge case once Firebird's actual FK-validity rules were checked directly.
