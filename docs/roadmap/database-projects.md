# Firebird Database Projects (schema-as-code)

**Inspired by**: [vscode-mssql](https://github.com/microsoft/vscode-mssql)'s SQL Database Projects extension (offline `.sqlproj` authoring, build, and publish) and DACPAC/BACPAC deploy/extract.

## Current state in Firebird Studio

**Phases 1 and 2 are done.** Right-click a database ‚Üí **Extract Database Project...** writes the connected schema out as one `.sql` file per object; **Build Database Project...** (Command Palette ‚Äî it operates on a folder, not a live connection) concatenates an extracted project into one reviewable deploy script:

- `src/database-projects/project-model.ts` ‚Äî pure DDL-reconstruction/manifest logic (no vscode/Driver dependency, unit-tested like `schema-graph.ts`/`openapi-spec.ts`, 23 tests): `buildTableCreateDDL()` reconstructs a `CREATE TABLE` from the same `SchemaColumn[]` model the Schema Designer already assembles (columns, `NOT NULL`, `DEFAULT`, a trailing `PRIMARY KEY (...)` clause); `buildForeignKeyDDL()` builds an `ALTER TABLE ... ADD CONSTRAINT ... FOREIGN KEY` per relationship; `buildProcedureCreateDDL()`/`buildTriggerCreateDDL()`/`buildViewCreateDDL()` wrap each object's already-complete `RDB$PROCEDURE_SOURCE`/`RDB$TRIGGER_SOURCE`/`RDB$VIEW_SOURCE` text in a `CREATE OR ALTER` header ‚Äî confirmed these source columns already contain everything after the object's name (params/`RETURNS`/`AS`/body for procedures, `ACTIVE...AS`/body for triggers, just the `SELECT` for views) by checking how `NodeProcedure#editProcedure()`/`NodeTrigger#editTrigger()`/`NodeView#editView()` already reconstruct a runnable `ALTER` from the same columns; `buildProjectFiles()` assembles everything (tables, then `foreign-keys.sql`, then views/procedures/triggers/generators) plus a `firebird.project.json` manifest recording that same dependency-safe order, which the Build step just replays.
- `src/database-projects/index.ts` ‚Äî `runExtractProject()` fetches tables+FKs (`getSchemaColumnsQuery()`/`getForeignKeysQuery()`, same as the Schema Designer) plus three new "all objects, one round trip" queries (`getAllViewSourcesQuery()`/`getAllProcedureSourcesQuery()`/`getAllTriggerSourcesQuery()`, added to `src/shared/queries.ts` alongside the existing single-name versions) via one `Driver.runBatch()` call, then writes every file to the chosen folder. `runBuildProject()` reads an existing project's manifest, concatenates its files in the recorded order, and opens the result in an editor for review ‚Äî **never executed automatically**, matching this extension's existing "always show generated DDL before running it" convention (mock data, table designer, flat file import).
- Domains, roles, exceptions, and users are **not** extracted (see "Explicitly deferred" below) ‚Äî reconstructing valid DDL for each is its own, separate scope (now covered instead by the "Generic Script as Create/Drop" item, one object at a time).
- **NUMERIC/DECIMAL precision fixed** (was a disclosed gap here, now resolved as part of implementing "Generic Script as Create/Drop"): `getSchemaColumnsQuery()`/`tableInfoQuery()` now also select `RDB$FIELD_SUB_TYPE`/`RDB$FIELD_PRECISION`/`RDB$FIELD_SCALE`, confirmed directly against a live Firebird server (sub_type 1 = NUMERIC, 2 = DECIMAL, scale is negative). `columnTypeToDDL()` uses these to emit `NUMERIC(p,s)`/`DECIMAL(p,s)` instead of the bare underlying `INTEGER`/`BIGINT`/`DOUBLE` type whenever they're present ‚Äî a `NUMERIC(9,2)` column now round-trips correctly through Extract.

The pre-existing state this builds on: `src/schema-diff/schema-diff.ts` (262 lines) already did the hard part of this feature's foundation ‚Äî `fetchSchemaSnapshot()` reads a full schema from a live connection, `diffSchemas()` compares two snapshots, `renderDiffReport()` produces a **text report** ‚Äî but that only ran connection-vs-connection, ad hoc, and produced a human-readable diff, not an executable migration script or an on-disk project. Backup/restore (`gbak`/`nbackup` integration) already covers the DACPAC/BACPAC "package a whole database" use case reasonably well for Firebird (a `.fbk` backup *is* schema+data in one portable file), so this feature focuses on the **schema-as-code / versioned-migration** angle mssql's Database Projects extension targets, rather than re-implementing packaging `gbak` already does.

### Explicitly deferred (not done)

- **Phase 3 ‚Äî Publish/migrate**: no `ALTER TABLE`/`CREATE OR ALTER PROCEDURE` migration script generated by diffing a built project against a *live* target connection ‚Äî this remains "generate a deploy script from what I extracted," not "compute what changed and migrate it." Per the design doc's own risk callout, this is the single biggest net-new piece of logic (safe, executable `ALTER` generation ‚Äî e.g. a column type change needing a temp-column-and-copy dance, or FK drop/recreate ordering) and deserves its own careful design pass rather than being folded in here.
- **Workspace-level project binding**: no `.vscode/firebird.json`-style "this workspace's project folder + connection" config; Extract/Build always prompt for a folder.
- Domains, roles, exceptions, users (extracting a whole database's worth of them into project files) ‚Äî one-at-a-time DDL for these object types now exists via "Generic Script as Create/Drop" instead.

## Proposed feature

1. **Project structure**: a folder (e.g. `firebird-project/`) containing one `.sql` file per object (tables, views, procedures, triggers, generators, domains), plus a manifest (`firebird.project.json`) listing target Firebird version and object ordering/dependencies.
2. **Extract**: given a live connection, walk the same metadata queries the tree provider uses and write out one file per object ‚Äî effectively "reverse-diff" a live schema into a project. This can share code with `fetchSchemaSnapshot()`.
3. **Build**: concatenate/order the project's files into a single deployable script, validating dependency order (tables before FKs that reference them, etc.) ‚Äî the ordering logic is the main new piece; `schema-diff.ts` doesn't need this today since it only compares two already-existing schemas.
4. **Publish/migrate**: diff the built project against a target connection's live schema (reusing `diffSchemas()` almost as-is, just with one side being "project snapshot" instead of "connection B snapshot") and generate an executable migration script (`ALTER TABLE ...`, `CREATE OR ALTER PROCEDURE ...`) instead of today's human-readable-only report. This is the single biggest net-new piece of logic: `renderDiffReport()` currently formats differences for reading, not for re-execution.
5. Surface as commands: `firebird.project.extract`, `firebird.project.build`, `firebird.project.publish`, discoverable from a connection's context menu (same `contributes.menus."view/item/context"` pattern used for `firebird.schemaDiff`, see `package.json`).

## Technical notes

- Reuses more existing code than any other item on this list ‚Äî `schema-diff.ts`'s snapshot/diff engine is directly extensible rather than needing a parallel implementation.
- The riskiest new logic is generating *safe, executable* ALTER statements from a diff (e.g. handling a column type change that requires a temp-column-and-copy dance in some cases, or FK drop/recreate ordering) ‚Äî this deserves its own careful design pass once started, likely with a "preview, then confirm" step before ever running against a real database, matching the "always show generated SQL for review" convention already established for DDL-generating features (mock data, table designer).
- Workspace-level config already has a placeholder ROADMAP item ("Workspace-level database configuration (`.vscode/firebird.json`)") ‚Äî that setting is a natural home for "which project folder + which connection is this workspace's project bound to."
