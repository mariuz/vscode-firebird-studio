# SSH Tunneling for Connections

**Inspired by**: [vscode-pgsql](https://marketplace.visualstudio.com/items?itemName=ms-ossdata.vscode-pgsql)'s SSH tunneling support for reaching a database that only listens on a private/internal network.

## Current state in Firebird Studio

**Done.** `ConnectionOptions.sshTunnel` (`src/interfaces/connection-options.interface.ts`) carries `{host, port, user, authMethod: 'password'|'privateKey'|'agent', privateKeyPath?}`; the Add Connection wizard (`src/shared/connection-wizard.ts`) has a new step (after WireCrypt, non-embedded connections only) that collects it, plus the SSH password/passphrase, stored via `CredentialStore.storeSshPassword()`/`getSshPassword()`/`deleteSshPassword()` — the exact same SecretStorage pattern as the database password, never written to `globalState`.

- **`src/shared/ssh-tunnel.ts`** — `openSshTunnel()` uses [`ssh2`](https://www.npmjs.com/package/ssh2) (a new real dependency; see "Technical notes" below) to open a local forwarded port (bound to an OS-assigned ephemeral port, `127.0.0.1:0`) that tunnels to the real Firebird host/port through the SSH server. `buildConnectConfig()` maps `authMethod` to `ssh2`'s `ConnectConfig` (password, or a `readFileSync`'d private key + optional passphrase, or `SSH_AUTH_SOCK`/`pageant` for agent auth) — pure and unit-tested (13 tests) without needing a real SSH server.
- **`SshTunnelClient`** (same file) — a `ClientI` decorator, the same shape as `connection-pool.ts`'s `PooledClient`, keyed by `connectionOptions.id`: `createConnection()` opens (or reuses) a tunnel and rewrites `host`/`port` to the local forwarded port before delegating to the wrapped client, so a saved connection's tunnel is established once and reused across every subsequent connection attempt rather than re-opened per query. Always wraps outermost in `Driver.setClient()` (`this.client = new SshTunnelClient(pooling ? new PooledClient(...) : rawClient)`) — a pure passthrough for any connection without `sshTunnel` set, so this costs nothing when unused.
- **Two pre-existing gaps fixed while wiring this in** (both invisible until something actually needed positional-parameter binding to work through every layer):
  - `Driver.getQueryPlan()`'s `this.client instanceof NativeClient` check would have always been false once `this.client` became permanently wrapped in `SshTunnelClient` — fixed via a new `Driver.unwrapClient()` helper (and a matching `unwrap()` accessor added to both `PooledClient` and `SshTunnelClient`) that walks past any decorator to find the real driver instance.
  - `NativeClient.getQueryPlan()` connects internally (`this.createConnection(...)`), bypassing the `SshTunnelClient` wrapper entirely — fixed by resolving the connection options through `SshTunnelClient.resolveConnectionOptions()` (a new public method, also used internally by `createConnection()`) *before* calling into it, so a tunneled native-driver EXPLAIN PLAN request is still routed through the tunnel.

### Explicitly deferred (not done)

- **Live end-to-end verification against a real SSH server**: this sandbox has `sshd` installed but not running, and standing one up for a self-test would have required writing to the sandbox user's real `~/.ssh/authorized_keys` — correctly blocked by this session's safety tooling as an unauthorized credential-store modification, and not worth pursuing around. Verified instead via 13 unit tests on the pure/deterministic logic (`buildConnectConfig()`'s per-auth-method config building, `SshTunnelClient`'s passthrough-when-unconfigured behavior against a fake `ClientI`) plus a full real-Extension-Development-Host regression run (48 passing, 0 failing) confirming the bundled `ssh2` dependency doesn't break activation or any existing connection path.
- **`.vscode/firebird.json` workspace connections**: no `sshTunnel` support there — only the interactive Add Connection wizard collects it. A workspace-committed tunnel config would also need to decide where the SSH credential itself lives (it can't go in the committed JSON file), which is its own design question.
- **Per-connection tunnel teardown**: a tunnel opens on first use and stays open until `Driver.shutdown()` (extension deactivation) or the driver's client is replaced (`Driver.setClient()`, e.g. toggling `firebird.useNativeDriver`) — removing/dropping the saved connection itself doesn't proactively close its tunnel, matching how `PooledClient` already doesn't tear down pooled connections on connection removal either.

## Technical notes

- **Dependency trade-off — this was the one place hand-rolling wasn't reasonable**, and this is what actually shipped: SSH's transport/auth/channel protocol is a poor fit for this codebase's usual "hand-roll it, it's small" approach (used successfully for the CSV parser, chart SVG builder, connection-string parser) — it needs a real, security-sensitive protocol implementation. Went with **`ssh2`** as a new runtime dependency, consistent with already accepting `@modelcontextprotocol/sdk` for the MCP server rather than hand-rolling JSON-RPC framing (the shell-out-to-system-`ssh`-binary alternative considered in the original draft of this doc was rejected: weaker Windows guarantees, and more moving parts — `known_hosts`, agent forwarding config — to diagnose when it fails).
- **`.node` files (`ssh2`'s optional native crypto acceleration, `cpu-features`) can't be esbuild-bundled** — esbuild has no loader for them. Fixed by adding `--external:*.node` to the `esbuild-base` script (alongside the pre-existing `--external:node-firebird-native-api`): esbuild leaves those specific `require()` calls untouched rather than trying to inline them, and `ssh2`'s own `crypto.js` already wraps that require in a bare `try {} catch {}` with no rethrow — it falls back to pure-Node `crypto` (confirmed by reading `node_modules/ssh2/lib/protocol/crypto.js` directly) when the native binary isn't present, which is always the case here since it's never shipped. Unlike `node-firebird-native-api`, `ssh2`'s own JS code (plus its two runtime deps, `asn1`/`bcrypt-pbkdf`) bundles inline into `extension.js` normally — no `.vscodeignore` carve-out needed, no unpacked `node_modules/ssh2` in the packaged VSIX at all.
- Private key auth uses a file picker (`vscode.window.showOpenDialog`) for the key path, plus an optional passphrase stored the same way as the SSH password.
- The tunnel's local ephemeral port is picked fresh per tunnel (bind to port 0, read back the OS-assigned port via `server.address()`) rather than a fixed port, to avoid collisions when multiple tunneled connections are open at once.
