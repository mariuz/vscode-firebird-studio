# SSH Tunneling for Connections

**Inspired by**: [vscode-pgsql](https://marketplace.visualstudio.com/items?itemName=ms-ossdata.vscode-pgsql)'s SSH tunneling support for reaching a database that only listens on a private/internal network.

## Current state in Firebird Studio

**Done.** `ConnectionOptions.sshTunnel` (`src/interfaces/connection-options.interface.ts`) carries `{host, port, user, authMethod: 'password'|'privateKey'|'agent', privateKeyPath?}`; the Add Connection wizard (`src/shared/connection-wizard.ts`) has a new step (after WireCrypt, non-embedded connections only) that collects it, plus the SSH password/passphrase, stored via `CredentialStore.storeSshPassword()`/`getSshPassword()`/`deleteSshPassword()` ‚Äî the exact same SecretStorage pattern as the database password, never written to `globalState`.

- **`src/shared/ssh-tunnel.ts`** ‚Äî `openSshTunnel()` uses [`ssh2`](https://www.npmjs.com/package/ssh2) (a new real dependency; see "Technical notes" below) to open a local forwarded port (bound to an OS-assigned ephemeral port, `127.0.0.1:0`) that tunnels to the real Firebird host/port through the SSH server. `buildConnectConfig()` maps `authMethod` to `ssh2`'s `ConnectConfig` (password, or a `readFileSync`'d private key + optional passphrase, or `SSH_AUTH_SOCK`/`pageant` for agent auth) ‚Äî pure and unit-tested (13 tests) without needing a real SSH server.
- **`SshTunnelClient`** (same file) ‚Äî a `ClientI` decorator, the same shape as `connection-pool.ts`'s `PooledClient`, keyed by `connectionOptions.id`: `createConnection()` opens (or reuses) a tunnel and rewrites `host`/`port` to the local forwarded port before delegating to the wrapped client, so a saved connection's tunnel is established once and reused across every subsequent connection attempt rather than re-opened per query. Always wraps outermost in `Driver.setClient()` (`this.client = new SshTunnelClient(pooling ? new PooledClient(...) : rawClient)`) ‚Äî a pure passthrough for any connection without `sshTunnel` set, so this costs nothing when unused.
- **Two pre-existing gaps fixed while wiring this in** (both invisible until something actually needed positional-parameter binding to work through every layer):
  - `Driver.getQueryPlan()`'s `this.client instanceof NativeClient` check would have always been false once `this.client` became permanently wrapped in `SshTunnelClient` ‚Äî fixed via a new `Driver.unwrapClient()` helper (and a matching `unwrap()` accessor added to both `PooledClient` and `SshTunnelClient`) that walks past any decorator to find the real driver instance.
  - `NativeClient.getQueryPlan()` connects internally (`this.createConnection(...)`), bypassing the `SshTunnelClient` wrapper entirely ‚Äî fixed by resolving the connection options through `SshTunnelClient.resolveConnectionOptions()` (a new public method, also used internally by `createConnection()`) *before* calling into it, so a tunneled native-driver EXPLAIN PLAN request is still routed through the tunnel.
- **Verified against a real SSH server.** The original pass here only had 13 unit tests on the pure/deterministic logic plus a full Extension-Development-Host regression run, because standing up a real `sshd` in that sandbox would have meant writing to the sandbox user's actual `~/.ssh/authorized_keys` ‚Äî correctly blocked as an unauthorized credential-store modification. Once back in an environment where that restriction didn't apply, verified live instead: a throwaway `sshd` run as the unprivileged user on a scratch port (its own generated host key, its own `authorized_keys` file under a scratch directory ‚Äî nothing under the real `~/.ssh` touched at all), with a real Firebird server as the tunneled target. Confirmed, against that real server: `openSshTunnel()` opens a working local forwarded port for private-key auth and a raw socket through it actually reaches the target; `close()` really tears the listener down; a wrong private key, a password attempt against a pubkey-only server, and an unreachable SSH host all reject cleanly instead of hanging. One level up, `SshTunnelClient` wrapping the real `NodeClient` (the exact decorator `Driver.setClient()` installs) ran a real SQL query through the tunnel and got the right row back, reused the same tunnel across a second `createConnection()` call for the same connection id without reopening it, and `shutdown()` closed it cleanly. No bugs found ‚Äî unlike the Live Profiler's phase 5 (`docs/roadmap/live-profiler.md`), which is what live verification is for.

### `.vscode/firebird.json` workspace connections

- **`parseWorkspaceSshTunnel()`** (`src/shared/workspace-config.ts`, pure, unit-tested) parses an optional `sshTunnel: {host, port, user, authMethod, privateKeyPath}` object per entry ‚Äî structural, non-secret config only, the same fields the interactive wizard already collects. The design question the roadmap originally flagged ("where does the SSH credential live, since it can't go in the committed JSON file") turned out to already have an answer once looked at closely: `CredentialStore.getSshPassword()`/`SshTunnelClient` were already keyed purely by `connectionOptions.id` (`src/shared/ssh-tunnel.ts`'s `openTunnelFor()`), with no assumption about *where* that `ConnectionOptions` object came from ‚Äî and `workspaceConnectionId()` already gives every workspace connection a stable, deterministic id, the exact same "deterministic id, credential lives in SecretStorage regardless of where the config itself came from" split the DB password already uses for workspace connections. So no changes were needed to the tunnel-opening path at all ‚Äî only to *parsing* the structural config, plus a new way to *set* the credential (below).
- A malformed/incomplete `sshTunnel` (bad `authMethod`, missing `host`/`port`/`user`, `privateKey` with no `privateKeyPath`) is a warning + "ignore just the tunnel," not a reason to drop the whole connection entry ‚Äî same "one bad field doesn't take down the whole entry" policy `parseWorkspaceConfig()` already applies to `wireCrypt`/`password`. A `password`/`passphrase` field nested inside a committed `sshTunnel` object is rejected the same way the top-level `password` field already is ‚Äî SSH credentials must never be committed either. An `sshTunnel` on an `embedded` entry is also dropped (with a warning) ‚Äî an embedded/local-file database has no network host to tunnel to, matching the wizard's own "non-embedded connections only" restriction on this step.
- **New `firebird.database.setSshTunnelPassword` command** ("Set SSH Tunnel Password" on a database's right-click menu) ‚Äî `NodeDatabase.setSshTunnelPassword()` mirrors the existing `setPassword()`/"Set Connection Password" exactly (same `CredentialStore` SecretStorage mechanism, same unconditional availability on every database node), and is genuinely the only way to set an SSH credential for a workspace connection (there's nowhere else it could go) *and* the only way to change one for any connection post-creation ‚Äî previously the wizard only ever collected an SSH password/passphrase once, at connection-creation time, with no way to rotate it short of deleting and re-adding the whole connection.
- **Verified against a real SSH server and a real Firebird server** (same methodology as the original phase, not just unit tests on the parser): a throwaway `sshd` (its own scratch host key, its own `authorized_keys` ‚Äî nothing under the real `~/.ssh` touched), a real Firebird 6.0 scratch database, and a real `.vscode/firebird.json` on disk declaring a `privateKey`-auth `sshTunnel` plus a (deliberately included, to prove it's rejected) committed `password` field. Confirmed: `loadWorkspaceConnections()` reads the real file and returns a connection whose `sshTunnel` matches the committed config, whose top-level `password` never surfaced, and whose `id` is the exact same deterministic id a real "Set SSH Tunnel Password" run would target; storing a credential via `CredentialStore.storeSshPassword()` under that id and reading it back round-tripped correctly; and ‚Äî the actual point of this phase ‚Äî `SshTunnelClient` opened a real tunnel through the real `sshd` using that workspace-sourced `ConnectionOptions` and ran a real query against the real Firebird server through it, with a second `createConnection()` call for the same connection id reusing the already-open tunnel rather than opening a second one. No changes were needed to `ssh-tunnel.ts` itself ‚Äî this confirms the "already id-based, already source-agnostic" design read above was correct, not just plausible.
- **Found, but out of scope to fix here**: a bare `SELECT <varchar-column> FROM <table>` (no expression/concatenation wrapping it) against this particular scratch table triggered a pre-existing `node-firebird` response-parsing error ("expected length 12, actual 24") ‚Äî reproduced identically with zero SSH tunnel involved (confirmed by attaching directly), so it's a `node-firebird`/Firebird 6.0 wire-protocol quirk unrelated to tunneling, not something this phase's code touches. Worked around in the verification script with `SELECT MSG || '' AS MSG ...`; not chased further since it's a vendored dependency issue outside this roadmap item's scope.

### Explicitly deferred (not done)

- **Per-connection tunnel teardown**: a tunnel opens on first use and stays open until `Driver.shutdown()` (extension deactivation) or the driver's client is replaced (`Driver.setClient()`, e.g. toggling `firebird.useNativeDriver`) ‚Äî removing/dropping the saved connection itself doesn't proactively close its tunnel, matching how `PooledClient` already doesn't tear down pooled connections on connection removal either.

## Technical notes

- **Dependency trade-off ‚Äî this was the one place hand-rolling wasn't reasonable**, and this is what actually shipped: SSH's transport/auth/channel protocol is a poor fit for this codebase's usual "hand-roll it, it's small" approach (used successfully for the CSV parser, chart SVG builder, connection-string parser) ‚Äî it needs a real, security-sensitive protocol implementation. Went with **`ssh2`** as a new runtime dependency, consistent with already accepting `@modelcontextprotocol/sdk` for the MCP server rather than hand-rolling JSON-RPC framing (the shell-out-to-system-`ssh`-binary alternative considered in the original draft of this doc was rejected: weaker Windows guarantees, and more moving parts ‚Äî `known_hosts`, agent forwarding config ‚Äî to diagnose when it fails).
- **`.node` files (`ssh2`'s optional native crypto acceleration, `cpu-features`) can't be esbuild-bundled** ‚Äî esbuild has no loader for them. Fixed by adding `--external:*.node` to the `esbuild-base` script (alongside the pre-existing `--external:node-firebird-native-api`): esbuild leaves those specific `require()` calls untouched rather than trying to inline them, and `ssh2`'s own `crypto.js` already wraps that require in a bare `try {} catch {}` with no rethrow ‚Äî it falls back to pure-Node `crypto` (confirmed by reading `node_modules/ssh2/lib/protocol/crypto.js` directly) when the native binary isn't present, which is always the case here since it's never shipped. Unlike `node-firebird-native-api`, `ssh2`'s own JS code (plus its two runtime deps, `asn1`/`bcrypt-pbkdf`) bundles inline into `extension.js` normally ‚Äî no `.vscodeignore` carve-out needed, no unpacked `node_modules/ssh2` in the packaged VSIX at all.
- Private key auth uses a file picker (`vscode.window.showOpenDialog`) for the key path, plus an optional passphrase stored the same way as the SSH password.
- The tunnel's local ephemeral port is picked fresh per tunnel (bind to port 0, read back the OS-assigned port via `server.address()`) rather than a fixed port, to avoid collisions when multiple tunneled connections are open at once.
