# Graphical Query Plan Visualizer

**Inspired by**: [vscode-mssql](https://github.com/microsoft/vscode-mssql)'s Query Plan Visualizer (interactive node navigation over the SQL Server execution plan) and its Estimated/Actual Plan toggle commands, and [vscode-pgsql](https://marketplace.visualstudio.com/items?itemName=ms-ossdata.vscode-pgsql)'s Query Plan Visualizer (four synchronized views — node graph, icicle chart, sortable table, and raw source — plus importing a saved plan file with no live connection, and a "Analyze with Copilot" action).

## Current state in Firebird Studio

**All 7 phases are done.** `Driver.getQueryPlan()` (`src/shared/driver.ts`) still returns plain text — that's unchanged, and `firebird.explainPlan` still dumps it into a plaintext editor exactly as before — but there's now two graphical ways to view the same plan (a standalone panel, and a per-statement result-view tab):

- `src/shared/plan-parser.ts` — `parsePlan(planText): PlanNode[]` parses Firebird's legacy `PLAN (...)` syntax into a tree (or forest — a statement with subqueries produces multiple top-level blocks). **The grammar was reverse-engineered against a real Firebird 3.0 server** (a scratch database created and queried directly via `isql-fb` in this environment, not just recalled from documentation), which corrected two wrong assumptions this doc originally made:
  1. `JOIN` doesn't nest each participant as its own sub-plan (`JOIN (PLAN(...), PLAN(...))`) — it's a **flat**, comma-separated list of scans, regardless of how many tables are joined: `JOIN (A NATURAL, B INDEX (IB), C INDEX (IC))`.
  2. `SORT` isn't only ever wrapped around a `JOIN` — it can wrap a flat list of scans directly (seen from a `UNION` query) *or* a nested `JOIN (...)` (seen from an `ORDER BY` on a joined query that couldn't use an index to satisfy it). Wrapper nodes (`JOIN`/`HASH`/`MERGE`/`SORT`) genuinely recurse into either scans or other wrappers.
  
  `HASH` was directly observed too (a hash join); `MERGE` wasn't triggered by the available test data but shares the identical `KEYWORD (item, item, ...)` shape as the other three in every case that was observed, so it's parsed the same way. See the file's header comment for the full reasoning and the exact captured strings this was built against.
- `src/query-plan-view/` — a new webview (`QueryPlanView`, following the same scaffolding as `schema-designer`/`result-view`) that renders the parsed tree as a node diagram: a classic layered-tree layout (scans are leaves, wrapper nodes branch), pan/zoom/fit, click a node for a detail panel (access method, index name(s)), and a "Raw Text" toggle showing the underlying `PLAN (...)` string. New command `firebird.showEstimatedPlan` (keybinding `Ctrl+Alt+Shift+E` / `Cmd+Alt+Shift+E`, mirroring `firebird.explainPlan`'s `Ctrl+Alt+E`) opens it for the active editor's query, same resolution behavior (`Driver.getQueryPlan()` resolves from the active editor/active connection when not given explicit args). When the pure-JS driver's non-machine-readable fallback text is detected (checked by prefix before even attempting to parse), shows a clear "enable the native driver" message instead of a parse error.

### Phase 5 — table view and "import a saved plan"

- **Sortable table view**: a "Table View" toolbar toggle switches the same parsed tree from the node diagram to a flat, one-row-per-node `<table>` (`flattenBlocks()` in `app.js` walks the tree depth-first) with columns Node/Table/Access Method/Index(es)/Depth; clicking a column header sorts by it (numeric compare for `#`/Depth, case-insensitive string compare otherwise), clicking again reverses direction. No new layout code or extension-host round-trip — it's a second renderer over the exact same `blocks` data the diagram already has.
- **Cross-view selection**: clicking a node in either the diagram or the table selects it and shows the same detail panel; selection now tracks the stable underlying `PlanNode` (`selectedNode`) rather than an ephemeral layout-tree object, which also fixes a latent bug in the original diagram-only selection — `render()` rebuilds fresh layout objects on every call, so comparing against the previous call's layout-tree node (`selectedLayoutNode === layout`) could never actually match, meaning the `fb-selected` highlight silently never applied. Found while wiring the table view to share selection state with the diagram, not through separate investigation.
- **Import a saved plan**: a new "Import Plan" toolbar button (`QueryPlanView.importFromFile()`) opens a file picker, reads the file via the same `fs/promises` `readFile()` pattern `flat-file-import/index.ts` already uses, and feeds it through the exact same `parseAndSend()` path a live fetch uses (fallback-text detection, `parsePlan()`, error reporting) — so an imported plan (e.g. one pasted from `firebird.explainPlan`'s plaintext output, or `isql`'s `SET PLANONLY ON`) behaves identically to a freshly-fetched one, with no live connection needed.
- Still not done from phase 5: a "source" view (vscode-pgsql's fourth synchronized view, distinct from this table view) wasn't in the original phase description in enough detail to scope separately from the table view — no action taken here.
- **Verified**: a one-off Node harness (not committed, same convention as the tree-layout math below) drove `flattenBlocks()`/`sortRows()` against the real `SORT (JOIN (D NATURAL, E INDEX (FK_EMP_DEPT)))` fixture from `plan-parser.test.ts` — confirmed depth-first row order/depths, correct index-detail extraction, and both ascending/descending sort (string and numeric columns).

### Phase 4 — result-view tab integration

- **A "Query Plan" toolbar toggle on every batch result** (`src/result-view/htmlContent/js/app.js`'s `buildEditableTable()`), right alongside the existing "🤖 Analyze" button and gated on the same condition — it only appears when the SQL that produced this particular result set is known (batch results from `firebird.runQuery`; the single/legacy `display()` path, used by predefined actions like Show Table Info, doesn't track it, matching Analyze's existing restriction). Clicking it requests the plan for *that specific statement's* SQL via the existing `requestFromExtension()`/`resolveRequest()` round-trip `getPrimaryKey`/`applyChanges` already established, rather than inventing a second messaging pattern.
- **`ResultView.handleGetQueryPlan()`** (`src/result-view/index.ts`) fetches via `Driver.getQueryPlan(sql)` and interprets the result through the exact same `interpretPlanText()` helper `QueryPlanView` uses (see below) — so the tab and the standalone panel render identical errors for identical inputs. Results are cached per statement SQL text (`planCache`, cleared on every new `display()`/`displayBatch()`), so switching tabs back and forth doesn't re-fetch.
- **`src/shared/plan-parser.ts` gained `interpretPlanText()`** (plus its `PLAN_FALLBACK_PREFIXES` constant), extracted from what was previously `QueryPlanView`'s private `parseAndSend()` — the fallback-text detection, `parsePlan()` call, and error-message shape are identical for both consumers, so this is one tested implementation instead of two independently-drifting copies. `QueryPlanView` was refactored to call it too.
- **`src/result-view/htmlContent/js/plan-view.js`** is a new, self-contained diagram/table renderer — an instance-scoped port of `query-plan-view/htmlContent/js/app.js`'s rendering logic (layout math, SVG diagram, pan/zoom/fit, table view + sort, detail panel, raw-text toggle), *not* a shared file the two webviews both load (this codebase's webviews are each self-contained with their own `htmlContent/` folder — no existing precedent for sharing JS/CSS across them, and a cross-folder relative script path would be fragile). The port was necessary, not just copy-paste: `query-plan-view/app.js` keeps all its state (`selectedNode`, `viewMode`, pan/zoom `view`, etc.) at module scope because that webview only ever shows one plan; `result-view` can have several "Query Plan" tabs open at once (one per batch statement), so every bit of that state had to move inside the closure `FirebirdPlanView.create(container)` returns. No "Refresh" or "Import Plan" controls here — refreshing would mean re-running the query (out of scope for a read-only tab) and importing a file makes sense for a connection-less standalone panel, not a tab already tied to one specific statement's result.
- Import/refresh aside, the rendered feature set (diagram, table view, selection, zoom/pan/fit, raw text) is the same as the standalone panel.

### Phase 6 — Copilot "Analyze" action

- **A "🤖 Analyze" toolbar button in both the standalone panel and the result-view "Query Plan" tab** — reuses `sendToModelAndShowResult()` and the `getSchemaBlock()`/`vscode.lm.selectChatModels()` machinery every other AI Query Action (`src/copilot/ai-query-actions.ts`) already established, via a new `buildAnalyzePlanMessages(sql, planText, schemaBlock)` prompt builder in `src/copilot/prompts.ts` and `runAnalyzePlanAction()`. The prompt asks the model to explain the plan in plain English, flag expensive operations (natural scans, unsupported sorts), and suggest concrete indexes — the raw `PLAN (...)` text is sent, not the parsed tree, since that's what a model actually reasons over well (the diagram/table are for the human, not the model).
- **Same delegation pattern as "🤖 Analyze" on query results**: both `QueryPlanView` and `ResultView` emit an `"analyzePlan"` event on their shared `QueryResultsView`/`EventEmitter` base rather than importing `src/copilot` directly; `extension.ts` wires both to one shared handler function.
- **`sql` is best-effort for the standalone panel, exact for the result-view tab**: `ResultView`'s "Query Plan" tab always knows precisely which statement it's showing a plan for (the tab only exists alongside that statement's own result set). `QueryPlanView.open()` is called with no explicit SQL in practice (`firebird.showEstimatedPlan` never passes one — `Driver.getQueryPlan()` resolves the active editor internally), so `QueryPlanView` tracks the raw text of whatever plan it most recently rendered successfully (`lastRawPlan`, cleared on error/re-open) and emits `sql: undefined`; `runAnalyzePlanAction()` then falls back to `getActiveEditorSql()`, the same "whatever's in the active SQL editor right now" resolution every other editor-facing AI action in this file already uses.
- **The Analyze button is only enabled once a real plan is showing** (not during loading, not on an error/empty result) — both webviews disable it in their error/empty branches and re-enable it only in the success branch, mirroring the existing "🤖 Analyze" button's SQL-known gating on the results side.
- **`plan-view.js` stays messaging-agnostic**: it doesn't know about `postMessage` at all (see its own docstring) — `create(container, { onAnalyze })` takes a plain callback invoked with the plan's raw text; `result-view/app.js` supplies the callback and is the only place that knows the actual message shape (`{command: "analyzePlan", data: {sql, plan}}`), keeping the messaging protocol in one file per webview.

### Phase 7 — icicle chart view

- **What it actually charts, and why it's not literal cost**: vscode-pgsql's icicle view charts a real optimizer's cost/time estimate per node. Firebird's legacy `PLAN (...)` text carries **no cost or row-count estimate at all** — it's purely "which access method, which index," nothing numeric. Rather than skip the view or fabricate numbers, bar width uses each node's share of the plan's total leaf scans — the same `countLeaves()` weight `layoutNode()` already computes for the diagram's own node spacing — as a structural proxy. Natural (unindexed) scans get the same warning color the diagram/table already use, since for this plan format "which scans are unindexed" is the closest available answer to "what looks expensive at a glance," which is what this phase was originally chartered to surface. This is called out explicitly in both webviews' tooltip on the "Icicle" button, not left for the user to discover the hard way.
- **A third view mode, not a fourth toggle button**: the previous binary "Table View"/"Diagram View" text-swap button (phase 5) didn't scale to three states cleanly, so it's now a 3-button group (Diagram/Table/Icicle) with `.active` highlighting the current mode — `#toolbar .view-mode-btn.active` / `.fb-plan-view-mode-btn.active`, same `viewMode`/`applyViewMode()` state machine as before, just with a third branch.
- **`icicleLayout(blocksArr)`** (added to both `query-plan-view/app.js` and `result-view/plan-view.js`, per this doc's established no-cross-webview-sharing convention — see phase 4's notes): walks the same parsed tree depth-first, returning one `{node, depth, x0, width}` segment per node, `x0`/`width` as 0–1 fractions of total chart width, a child's width split from its parent's proportional to `countLeaves(child) / totalChildLeaves`. Rendered as absolutely-positioned `<div>`s — plain HTML, not SVG, since a stacked-bar layout doesn't need SVG's pan/zoom/path machinery the diagram does.
- **Selection stays cross-view**: clicking an icicle segment sets the same `selectedNode` the diagram/table already share, so switching to the diagram or table afterward shows the same node highlighted/selected, and vice versa.
- **A real, unrelated CSS specificity bug found and fixed along the way**: `query-plan-view`'s toolbar buttons are styled via `#toolbar button.secondary { background: ...; }` — an ID selector. The new `.view-mode-btn.active { background: ...; }` (two plain classes) has *lower* specificity than an ID selector regardless of class count, so the active-mode highlight would have silently never applied, the same class of bug phase 5 fixed for click-selection. Fixed by scoping the rule as `#toolbar .view-mode-btn.active` instead. `result-view`'s equivalent (`.fb-plan-toolbar button`, no ID) wasn't affected — this was specific to the standalone panel's markup.
- **Verified**: a one-off Node harness drove `icicleLayout()` in *both* files against the real `SORT (JOIN (D NATURAL, E INDEX (FK_EMP_DEPT)))` fixture, checking the tiling invariants an icicle chart depends on for correctness — the root segment spans the full [0,1] width, every depth level's segments sum to exactly their parent's width (no gaps or overlaps), and the natural scan is correctly flagged — in both the module-scoped standalone webview and the instance-scoped `result-view` port.

### Explicitly deferred (not done)

- **Phase 3 — Actual vs. Estimated**: no `MON$STATEMENTS`/`MON$RECORD_STATS` overlay yet; the diagram only ever shows the estimated plan. Worth flagging for whoever picks this up next: Firebird's monitoring tables report I/O and record-access counters at the *statement* level, not per plan-node — there's no obvious `MON$` source for "this specific JOIN step did N index reads," so this phase may need a different data source entirely (e.g. Firebird's trace API) rather than being a straightforward overlay query, unlike phases 4/5/6/7 above which turned out to need no new data access at all. This is now the only remaining phase from the original design doc.

`src/test/plan-parser.test.ts` — 19 tests (15 for `parsePlan()`, 4 for `interpretPlanText()`), every fixture string captured verbatim from the real server (not invented), covering every scan method, all four wrapper keywords, nesting, mixed-case identifiers, multi-block plans, malformed input, and both fallback-prefix cases. Neither webview's rendering JS (`query-plan-view/htmlContent/js/app.js` or `result-view/htmlContent/js/plan-view.js`) has automated coverage, matching this repo's convention for webview inline JS — verified instead with one-off Node harnesses (not committed): the original tree-layout-math harness for `query-plan-view`; a `flattenBlocks()`/`sortRows()` harness for phase 5; for phase 4, one confirming `plan-view.js`'s `create()` produces fully independent instances (two simultaneous "Query Plan" tabs don't leak `selectedNode`/`blocks`/sort state into each other) plus a second harness driving the real `Driver.getQueryPlan()` fallback path (`extractTableNames`/`buildIndexMetadataQuery`/`renderIndexMetadataPlan` against a live Firebird 6.0 scratch database) through `interpretPlanText()`; for phase 6, a harness confirming the "🤖 Analyze" button starts disabled, is enabled only after a successful `show()`, invokes `onAnalyze` with exactly the plan's raw text on click, and is disabled again by a subsequent error result; and for phase 7, a harness checking `icicleLayout()`'s tiling invariants in both webviews against the same real fixture (see above). `buildAnalyzePlanMessages()` itself gained 3 unit tests in `src/test/copilot-prompts.test.ts` (alongside the existing Explain/Optimize/Analyze-Results prompt-builder tests — these are pure functions the mocked `vscode.LanguageModelChatMessage` makes testable without a real model) covering the SQL-known case, the SQL-unknown case (no empty ` ```sql ` block), and that the prompt asks about expensive operations/indexes. `runAnalyzePlanAction()` itself (like its siblings `runAiQueryAction()`/`runAnalyzeResultsAction()`) has no automated coverage — it depends on `vscode.lm.selectChatModels()`, which the unit-test mock doesn't model.

## Technical notes

- The native-driver requirement matters here: without it, there's no real machine-readable plan to parse (the pure-JS fallback is explicitly a heuristic string, not real Firebird plan syntax) — `interpretPlanText()` (`src/shared/plan-parser.ts`, used by both `QueryPlanView` and `ResultView`) detects this by checking the fallback text's known prefixes before attempting to parse, rather than surfacing a confusing parse error.
- Keep the plan parser and the rendering layer separate (parser is pure/testable; the webview is presentation-only) — matches this repo's stated convention of isolating SQL-parsing logic into pure functions. This paid off directly: the two grammar corrections above were found and fixed in the parser alone, with zero changes needed to the (not-yet-written-at-the-time) rendering code.
- If you need to re-derive real plan fixtures for further work on the parser (e.g. to finally trigger a genuine `MERGE` plan), a Firebird 3.0 server is reachable in this environment via `isql-fb`; `plan-parser.ts`'s file header documents the exact scratch-database setup used.
- The icicle chart (phase 7) turned out to need no cost/time data source at all, contrary to what this note originally assumed — see phase 7 above for what it charts instead and why.
