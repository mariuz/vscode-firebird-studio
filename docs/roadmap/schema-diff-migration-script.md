# Schema Diff: Generate a Runnable Migration Script

**Inspired by**: [vscode-mssql](https://github.com/microsoft/vscode-mssql)'s Schema Compare (GA 1.35.0) — "visually compare and script out schema differences" between two databases, producing an executable fix-up script from the comparison, not just a read-only report.

## Current state in Firebird Studio

**Done — via a different, simpler path than originally proposed.** `src/schema-diff/schema-diff.ts`'s existing `firebird.schemaDiff` command (`diffSchemas()`/`renderDiffReport()`) still only produces a read-only text report and is untouched by this work — the new capability lives alongside it, not inside it.

The original plan below (converting `schema-diff.ts`'s `SchemaDiffResult` into `database-projects/publish-model.ts`'s `PublishDiff` shape so `buildPublishScript()` could run against it) turned out to be the wrong approach once actually inspected: `SchemaDiffResult`'s underlying `SchemaSnapshot` is missing almost everything `PublishDiff`/`buildPublishScript()` need beyond bare table/column names and types — no view/procedure/trigger *source text* (`SchemaSnapshot` only ever fetches their *names*, via `getViewsQuery()`/`getStoredProceduresQuery()` — nothing like `getAllViewSourcesQuery()`), no foreign keys (never fetched at all by `fetchSchemaSnapshot()`), no domains, generators, exceptions, roles, or users. Writing the proposed "pure mapping function" would have meant either producing a script with those categories silently empty (wrong — a real schema diff between two live databases routinely differs in more than tables/columns) or first substantially enriching `schema-diff.ts`'s own snapshot-fetching to duplicate work `database-projects/index.ts`'s `fetchProjectSnapshot()` already does correctly.

That's the fix that was actually shipped: **`fetchProjectSnapshot(connectionOptions): Promise<ProjectInput>`** (`src/database-projects/index.ts`) already fetches everything `diffProjects()`/`buildPublishScript()` need directly from *any* live connection — it's exactly what `runPublishProject()`'s own target-side fetch already uses, just never before called on *both* sides of a comparison. A new `runGenerateMigrationScript()` in that same file does exactly that: prompts for a source and target connection (a new picker, deliberately independent of `firebird.schemaDiff`'s own — see below for why), an `includeDrops` choice (matching `runPublishProject()`'s own), calls `fetchProjectSnapshot()` on both, then `diffProjects()` + `buildPublishScript()` unchanged — zero new diffing or DDL-generation logic, zero conversion code. Wired to a new `firebird.schemaDiff.generateMigrationScript` command ("Generate Migration Script..." on a database's right-click menu, `second@5`), opening the result as a new SQL document for review, never executed automatically — matching `runPublishProject()`'s own "review it carefully, then run it yourself" framing exactly (this is the same code path, so it inherits the same behavior for free).

**Why an independent picker, not a shared one with `firebird.schemaDiff`**: the two commands' pickers are structurally identical (source pick, target pick with source excluded, password resolution for both) but were kept as separate, small, independently-owned blocks rather than extracted into a shared helper — refactoring the *already-shipped* `firebird.schemaDiff` command's working picker logic purely to deduplicate ~15 lines against a new command was judged not worth the risk to already-working code, consistent with this repo's general preference against incidental refactoring alongside a feature addition.

### Testing

The new command itself (QuickPick/`withProgress` orchestration) isn't independently tested — the same "VS Code dialog orchestration isn't unit-tested" boundary this repo already draws for `runPublishProject()`/`node-database.ts`'s wizard-style methods/`connection-wizard.ts` (see `docs/roadmap/connection-management-enhancements.md`'s own Testing section for the fullest statement of this). The actual underlying mechanism — `fetchProjectSnapshot()` on a live connection, `diffProjects()`, `buildPublishScript()` — needed **no new test coverage at all**, because `src/test/suite/database-project-publish-integration.test.ts` already exercises exactly this chain end-to-end against a real server: it fetches a `fetchProjectSnapshot()` "target" snapshot, evolves the live schema in place, fetches a second `fetchProjectSnapshot()` "source" snapshot of the *same connection*, diffs and scripts the two, *actually executes* the result, and re-verifies the target now matches — i.e., two live `fetchProjectSnapshot()` calls diffed against each other, precisely what `runGenerateMigrationScript()` does (the only difference being *which* two connections the two snapshots come from, which is invisible to `diffProjects()`/`buildPublishScript()` either way). `firebird.schemaDiff.generateMigrationScript`'s registration is covered by `src/test/suite/extension.test.ts`'s existing "every roadmap doc's key command(s) are actually registered" suite.
