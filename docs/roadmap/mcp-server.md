# MCP Server for Firebird

**Inspired by**: [vscode-pgsql](https://marketplace.visualstudio.com/items?itemName=ms-ossdata.vscode-pgsql)'s MCP server registration ("exposes PostgreSQL tools, including connection management, schema exploration, query execution, and query plan visualization, to supported AI-enabled hosts").

## Current state in Firebird Studio

**Phases 1-5 are done** (list_connections, get_schema, run_query, get_query_plan — read-only; run_write_query — opt-in write access).

Before writing any registration code, the API surface described below (written when this was purely directional) was validated against the actual installed `@types/vscode` (1.125.0 — notably newer than this extension's `engines.vscode: ^1.93.0` floor) and against VS Code's own MCP developer guide:

- **The registration model is not "handle MCP calls in-process."** `vscode.lm.registerMcpServerDefinitionProvider(id, provider)` only tells VS Code *where* an MCP server lives — `provider.provideMcpServerDefinitions()` returns `McpStdioServerDefinition`s (a command + args + env to spawn), and VS Code's own built-in MCP client spawns that as a **separate child process** and speaks MCP-over-stdio to it directly. The extension host is not on the request path once the process is running.
- This means the MCP tool implementations can't reuse `Driver`/`NodeClient`/`CredentialStore` at all — all three depend on the `vscode` module (`workspace`, `window`, `ExtensionContext.secrets`), which doesn't exist in a plain spawned Node process.
- A matching static declaration is required in `package.json`'s new `contributes.mcpServerDefinitionProviders` (`[{ "id": "firebird-mcp", "label": "Firebird MCP Server" }]`) alongside the runtime `registerMcpServerDefinitionProvider("firebird-mcp", ...)` call — registering only at runtime, without the matching package.json entry, fails.
- Guarded the same way `src/copilot/copilot-chat-participant.ts` guards `vscode.chat`: `src/mcp-server/index.ts#registerMcpServer()` checks `typeof vscode.lm !== 'undefined'` and `typeof vscode.lm.registerMcpServerDefinitionProvider === 'function'` at runtime before registering anything, so the extension still activates cleanly on a VS Code build without MCP support — no `engines.vscode` bump needed.

### What's implemented

- **`src/mcp-server/server.ts`** — the actual spawned subprocess (bundled to `out/mcp-server/server.js` via its own esbuild invocation, `npm run esbuild-mcp-server`, distinct from the main `extension.js` bundle since this runs as its own process). Built with `@modelcontextprotocol/sdk` (a new dependency — hand-rolling MCP's JSON-RPC framing/capability negotiation from scratch would be a poor trade next to a well-maintained official SDK, unlike the CSV parser or markdown-table renderer elsewhere in this codebase, which were small enough to reasonably hand-roll) and `zod` for tool input schemas (already a transitive dependency of the SDK; declared directly since `server.ts` imports it itself). Registers four tools:
  - `list_connections` — returns `{id, label, host, database}` for every connection the user explicitly exposed (never a password).
  - `get_schema(connectionId)` — connects via plain `node-firebird` (this subprocess doesn't bundle the native driver, so **embedded connections aren't supported here** — the same guard `NodeClient.createConnection()` already has), runs `getSchemaColumnsQuery()`/`getForeignKeysQuery()` (reused directly — genuinely dependency-free, as advertised), and returns `buildSchemaGraph()`'s result as JSON.
  - `run_query(connectionId, sql)` — **read-only, hard-coded, no opt-out**: `validateReadOnlyStatement()` (new, `src/shared/sql-analysis.ts`) rejects anything that isn't exactly one `SELECT`/`WITH ... AS (...) SELECT` statement — multi-statement scripts, INSERT/UPDATE/DELETE/DDL/EXECUTE BLOCK are all refused *before* a connection is even opened. Reuses `sql-splitter.ts`'s `splitStatements()` to detect "more than one statement," the same statement-boundary logic the rest of the extension already trusts.
  - `get_query_plan(connectionId, sql)` — same read-only validation as `run_query`, then the identical NodeClient-fallback index-metadata heuristic `Driver.getQueryPlan()` uses (this subprocess has no native driver, so Firebird's real `PLAN` output isn't available here either) — `extractTableNames()`/`buildIndexMetadataQuery()`/`renderIndexMetadataPlan()` were moved out of `driver.ts` into `src/shared/sql-analysis.ts` (dependency-free, no `vscode` import) specifically so this subprocess could reuse the exact same logic instead of a second, drifting copy; `driver.ts` re-exports `extractTableNames` so existing importers are unaffected.
  - **Critical implementation detail**: stdout is the MCP JSON-RPC stream itself — every log line in `server.ts` goes to `console.error` (stderr), never `console.log`, or it corrupts the protocol stream.
  - **Verified against a live server**, not just code review: a standalone script using the SDK's own `Client`/`StdioClientTransport` spawned the real built `server.js`, called all four tools against the live seeded database, confirmed `run_query` returns real rows for a SELECT, confirmed an INSERT is rejected with `isError: true`, and confirmed — by re-querying afterward — that the rejected INSERT never actually reached the database.
- **`src/mcp-server/index.ts`** (extension-host side) — `registerMcpServer(context)`, wired into `extension.ts#activate()`. `provideMcpServerDefinitions()` returns `[]` outright unless `firebird.mcp.enabled` is on (no separate "unregister the provider" path needed — an empty definition list is the idiomatic way to say "nothing available right now"). `resolveMcpServerDefinition()` is where credentials are actually resolved — per the API's own contract, "the extension may take actions requiring user interaction/authentication" specifically at this step — reading exposed connections from `context.globalState`, resolving each one's password via `CredentialStore.getPassword()`, and setting the spawned process's `env.FIREBIRD_MCP_CONNECTIONS` to a JSON array of `{id, label, host, port, database, user, password, role, embedded}` for **only the connections that opted in**.
- **Per-connection opt-in, not a blanket allowlist setting**: a new `ConnectionOptions.mcpExposed?: boolean`, toggled via **Toggle MCP Server Exposure** on a database's right-click menu (`NodeDatabase.toggleMcpExposure()`), guarded against workspace-sourced connections the same way `setConnectionColor()`/`setConnectionGroup()` already are. `list_connections`/`get_schema` only ever see connections with this explicitly set — turning `firebird.mcp.enabled` on exposes nothing by itself.
- **Credentials via env var, not argv or disk** — the same pattern `src/shared/isql-terminal.ts` already uses for a spawned local process (`ISC_PASSWORD` as an env var, not a visible command-line argument), applied here for consistency with an already-accepted precedent in this codebase, not a new security posture invented for this feature.

The pre-existing state this builds on: none — this was a genuinely new capability, worth distinguishing clearly from two things it's adjacent to but not the same as:

- **`src/copilot/copilot-chat-participant.ts`** (the `@firebird` chat participant, `/query`/`/optimize`/`/explain`/`/designSchema`) is scoped to *this extension's own* integration with VS Code's built-in Copilot Chat UI. It's not reachable by any other AI tool.
- **[`docs/roadmap/data-api-builder.md`](data-api-builder.md)** is about exposing a connected database's *data* (tables/views as CRUD endpoints) to *application developers* building software against it.
- **This feature** is about exposing this extension's own *database tooling* — connect, explore schema, run a query, get a plan — as callable tools to *any* MCP-compatible AI client or agent (Claude Desktop, Cursor, a custom agent script, VS Code's own Copilot in Agent mode), independent of whether that client is even running inside VS Code. Confirmed via vscode-pgsql's demo media: its Agent mode literally shows a tool-call line ("Visualizing schema for connection '...'") with a checkmark on completion — the AI agent is invoking the extension's own capabilities as MCP tools, not just chatting about SQL.

## Proposed feature

An MCP server, registered by this extension, exposing tools such as:

- `list_connections` — the saved connections currently known to the extension (name/host/database, never credentials).
- `get_schema(connectionId)` — serializes the connected database's schema, reusing `buildSchemaGraph()`/`schema-context.ts`'s existing serialization rather than inventing a second schema format.
- `run_query(connectionId, sql)` — executes via `Driver.runQuery()`/`runBatch()`, the same path every other feature in this extension already goes through (so it inherits query history logging, password resolution, etc. for free).
- `get_query_plan(connectionId, sql)` — wraps `Driver.getQueryPlan()`.

## Security model — the part to get right before anything else

This is the most consequential item on this whole roadmap from a trust standpoint: unlike the chat participant (which only ever runs inside this user's own VS Code, gated by whatever confirmation UI a command already shows), an MCP server is designed to be called by *external* processes — a different application entirely, potentially running with less human-in-the-loop oversight than a chat panel provides.

1. **Which connections are exposed at all** — ✅ resolved: `ConnectionOptions.mcpExposed`, an explicit per-connection opt-in via **Toggle MCP Server Exposure**. No saved connection is exposed by default, even with `firebird.mcp.enabled` on.
2. **Read vs. write** — ✅ resolved: `run_query`/`get_query_plan` remain hard-coded read-only with no opt-out or setting to relax it — `validateReadOnlyStatement()` rejects anything but a single `SELECT`/CTE before a connection is even opened. `run_write_query` (phase 5, below) is a separate, additional tool with its own separate opt-in (`mcpWriteEnabled`, on top of `mcpExposed`), its own single-statement INSERT/UPDATE/DELETE-only validation, its own explicit modal confirmation at the point access is granted (the only point in the whole path with a real VS Code UI available to ask), and its own audit trail — not a relaxation of `run_query`/`get_query_plan`'s guarantees.
3. **Credentials never cross the MCP boundary** — ✅ resolved, though "cross the MCP boundary" needed a more precise definition once the real architecture was understood: the MCP *client* (Claude Desktop, Cursor, VS Code's own Copilot Agent mode, ...) never sees a password — `get_schema`'s response is schema JSON only. But the resolved password *does* reach the spawned `server.ts` subprocess (via `env.FIREBIRD_MCP_CONNECTIONS`), because that subprocess is what actually opens the Firebird connection — there is no extension-host-side `Driver`/`CredentialStore` call it can make instead, since it isn't part of the extension host (see "Current state" above). This is the same exposure model `isql-terminal.ts` already accepts for a spawned local process (env var, not argv/disk), scoped further here by only ever including connections that opted in.

- Settings namespace: `firebird.mcp.enabled` (boolean, default `false`) gates the whole server. Write access has no global setting of its own — it's the per-connection `mcpWriteEnabled` flag (phase 5, below), not a workspace-wide toggle, matching `mcpExposed`'s own per-connection-opt-in shape rather than a blanket setting.

### Live refresh mid-session

**Done.** `onDidChangeMcpServerDefinitions` used to fire only when a `firebird.mcp.*` setting changed — toggling a connection's `mcpExposed` flag from the tree (**Toggle MCP Server Exposure**) wrote straight to `globalState` with no signal to the already-registered provider, so an already-running MCP client session needed a restart to see a newly-exposed (or newly-hidden) connection.

- `src/mcp-server/index.ts` now has a module-level `mcpExposureChangedEmitter`, plus two exports: `notifyMcpExposureChanged()` (fire it) and `onMcpExposureChanged` (the event itself, for anything — including a test — that wants to observe it directly rather than only fire-and-forget). `registerMcpServer()` relays it into the same `didChangeEmitter` the config-change listener already feeds, so both triggers end up calling `onDidChangeMcpServerDefinitions` the same way.
- `NodeDatabase.toggleMcpExposure()` (`src/nodes/node-database.ts`) calls `notifyMcpExposureChanged()` right after saving the flag, alongside the existing `firebirdTreeDataProvider.refresh()` call.
- Module-level (not threaded through `registerMcpServer()`'s local scope) so `toggleMcpExposure()` doesn't need a reference passed down from `extension.ts#activate()`, and calling it is a safe no-op if `registerMcpServer()` never ran (a VS Code build with no MCP API).
- **Verified in a real Extension Development Host** (`npm run test:vscode-host`, VS Code 1.129.0 — confirms `vscode.lm.registerMcpServerDefinitionProvider` is really present at runtime, not just in `@types/vscode`), via a new `src/test/suite/mcp-server-integration.test.ts`: constructs a real `NodeDatabase` against a fake-but-functional `ExtensionContext` (an in-memory `globalState`, no live Firebird connection needed since this path never touches the database), calls the real `toggleMcpExposure()`, and asserts `onMcpExposureChanged` fires exactly once and `mcpExposed` actually flips in the saved connection — plus a second test confirming a workspace-sourced connection's toggle is refused and fires nothing. VS Code core actually re-invoking `provideMcpServerDefinitions()` in response to the relayed event isn't independently observable from extension code (that's internal VS Code MCP-client machinery) — trusted the same as the pre-existing config-change trigger already was, per the documented `onDidChangeMcpServerDefinitions` contract.

### Phase 5 — opt-in write-query path

`run_query`/`get_query_plan` remain unconditionally read-only, exactly as before — this phase adds a **separate, additional** tool (`run_write_query`) rather than relaxing either of them, so nothing about the existing read-only guarantees changed.

- **A new, narrower opt-in on top of `mcpExposed`**: `ConnectionOptions.mcpWriteEnabled`, toggled via **Toggle MCP Server Write Access** on a database's right-click menu (`NodeDatabase.toggleMcpWriteAccess()`), requires `mcpExposed` to already be on for that connection — there's nothing to write-enable on a connection the MCP server can't even see. Turning `mcpExposed` *off* also clears `mcpWriteEnabled`, rather than leaving it dormant in storage: without this, re-enabling exposure later (a lower-stakes toggle with no confirmation of its own) would have silently reactivated a write grant set before exposure was last turned off, without ever going back through write access's own confirmation.
- **Where the one real confirmation actually happens**: enabling write access shows a modal `window.showWarningMessage` in VS Code, spelling out that any already-connected MCP client will be able to write, and that there is no per-query confirmation once this is on — because there's no way to add one. The spawned `server.ts` subprocess is the thing that actually runs a write, and it has no VS Code UI to show a dialog with (see the security model above) — so this toggle, on real VS Code UI, is the only point in the entire write-query path where genuine human confirmation is architecturally possible. Disabling write access needs no confirmation, matching every other permission-reducing toggle already in this codebase.
- **Scope: INSERT/UPDATE/DELETE only, not MERGE, not DDL.** `validateWriteStatement()` (`src/shared/sql-analysis.ts`, alongside the existing `validateReadOnlyStatement()`) rejects anything else — SELECT (that's `run_query`'s job), DDL, `EXECUTE BLOCK`, and multi-statement scripts are all refused before a connection is even opened, the same "validate before connecting" shape `run_query` already established. MERGE is a deliberate, disclosed exclusion for this first pass — ordinary CRUD writes cover the demand that prompted this phase; MERGE's combined insert-or-update-or-delete semantics can be added later if there's a concrete reason to.
- **`list_connections` now reports `writeEnabled` per connection**, so an MCP client can tell upfront which connections allow `run_write_query` without needing to try it first and get an error.
- **An audit trail, given the subprocess has no UI of its own to log through**: `run_write_query` appends one JSON line (`{timestamp, connectionId, sql, success, error?}`) to a file for *every* attempt — including a `writeEnabled: false` rejection, not just validation failures and real executions, matching the tool's own description's "every attempt, successful or not, is logged" claim. The path is handed to the subprocess via `FIREBIRD_MCP_AUDIT_LOG_PATH` (a file under `context.globalStorageUri`, resolved the same place/time `FIREBIRD_MCP_CONNECTIONS` already is). On the extension-host side, `startAuditLogWatcher()` watches that file's directory (not the file directly — it may not exist yet the first time a write-enabled session actually writes) and relays each new line into the extension's own `logger` (info for a success, error for a failure) as it's written, so a write an *external* MCP client made is actually visible in this VS Code window, not just recoverable by remembering to go find a log file. `fs.watch()` is inherently best-effort across platforms/filesystems, so a new **Show MCP Write Audit Log** command also opens the raw file directly as a durable fallback regardless of whether the live relay happened to catch every entry.
- A successful write whose statement had a `RETURNING` clause has its returned row(s) surfaced in the tool's response, the same way `node-firebird` already returns them — `run_write_query` doesn't special-case this, it's just what falls out of checking whether `db.query()`'s result is `undefined` (a plain DML statement) or populated (a `RETURNING` one).
- **Verified against a real Firebird 6.0 server**, using the same methodology as phases 1-4: a standalone script using the SDK's own `Client`/`StdioClientTransport` spawned the real built `server.js` with one write-disabled and one write-enabled connection, and confirmed: `list_connections` reports `writeEnabled` correctly for both; `run_write_query` rejects the write-disabled connection *before* ever opening a database connection; SELECT/DDL/multi-statement payloads are all rejected with the expected messages; a real INSERT and a real UPDATE on the write-enabled connection actually landed (confirmed by re-querying the database directly, independent of the MCP round trip); a write referencing a genuinely invalid column failed loudly with Firebird's own real error rather than silently; a write actually reaching the write-disabled connection's underlying database never happened; and the audit log ended up with exactly one entry per attempt (7 total: 1 flag-rejected, 3 validation-rejected, 2 successful, 1 execution-failed) with the right success/failure split and every field populated.
- New unit tests: `src/test/sql-analysis.test.ts` (15 tests) for `validateWriteStatement()`, mirroring `validateReadOnlyStatement()`'s existing coverage plus a MERGE-is-rejected case. `src/test/suite/mcp-server-integration.test.ts` (6 new Extension Development Host tests) drive the real `toggleMcpWriteAccess()` command path — including the modal confirmation, monkey-patching the real (unmocked) `vscode.window.showWarningMessage` for the duration of each test (there's no person available to click the dialog's button in a test run) — covering: refusing to enable write access before `mcpExposed`; a confirmed grant actually flipping the flag and firing the live-refresh event; a cancelled confirmation leaving it untouched; disabling needing no confirmation dialog at all; `toggleMcpExposure()` cascading to clear `mcpWriteEnabled` when exposure is turned off; and a workspace-sourced connection refusing the toggle entirely, same as the existing `mcpExposed` tests already establish.

### Explicitly deferred (not done)

- **Embedded connections**: `get_schema`/`run_query`/`get_query_plan` only support network connections — the spawned subprocess doesn't bundle the native driver embedded connections require (kept the subprocess bundle lean; this could be revisited if there's demand).
- **Automated tests for `server.ts`/`index.ts`'s core orchestration**: VS Code MCP API + subprocess spawning + stdio transport orchestration is hard to meaningfully unit-test, consistent with this repo's existing boundary for similar orchestration-heavy modules like `container-provisioning/index.ts`. The pure functions the tools are built from (`buildSchemaGraph`, the schema queries, and now `sql-analysis.ts`'s `validateReadOnlyStatement()`/`buildIndexMetadataQuery()`/`renderIndexMetadataPlan()`) all have their own unit coverage; the tools themselves were verified with a one-off script driving the real built subprocess through the SDK's own client (see above) rather than a permanent automated test. The live-refresh signal itself (above) *is* now covered, since unlike the tool-call plumbing it doesn't need a spawned subprocess to test meaningfully.
