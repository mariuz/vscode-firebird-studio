# Flat File Import Wizard

**Inspired by**: [vscode-mssql](https://github.com/microsoft/vscode-mssql)'s Flat File Import ("Import flat files (CSV, TXT) as new database tables using a guided wizard").

## Current state in Firebird Studio

**Phases 1, 2, 3, and 4 are done.** Right-click a database → **Import Flat File...** runs a guided QuickPick/InputBox wizard (no webview needed for this phase) that picks a CSV/TSV/JSON file, infers a schema, then either creates a new table or maps columns onto an existing one, and batch-inserts every row:

- `src/shared/flat-file-parser.ts` — parsing/inference module, unit-tested like `sql-formatter.ts` (91 tests): a hand-rolled RFC 4180-ish CSV/TSV parser (quoted fields, embedded delimiters/newlines, doubled-quote escaping, delimiter auto-sniffing between `,`/`\t`/`;`), a JSON array-of-objects parser, `inferColumnType()`/`inferSchema()` (INTEGER/BIGINT/NUMERIC(p,s)/BOOLEAN/DATE/TIMESTAMP/VARCHAR(n), sniffed from up to 200 sample rows, with nullability detection), `sanitizeIdentifier()` (Firebird 3.0's 31-char-safe identifier limit, since that's the floor across the FB3/4/5/6 CI matrix), DDL/INSERT builders, and (large-file streaming, below) a resumable chunked-stream parser plus a handful of real (but still vscode-free) file-reading helpers. No new npm dependency — `package.json` had no CSV parser before this, and one file's worth of RFC 4180 handling was small enough to hand-roll, per this repo's stated preference (matching the query-plan-view/schema-designer/profiler precedent of avoiding new dependencies).
- `src/flat-file-import/index.ts` (`runFlatFileImportWizard()`) — the wizard flow: pick a file → parse → choose a target (new table, or an existing one — see phase 3 below) → chunked (200 rows/`Driver.runBatch()` call) inserts under `vscode.window.withProgress`, using `Firebird.escape()` for value quoting (numeric/boolean columns get an unquoted literal from an actual JS number/boolean; everything else — including a row whose value doesn't actually match its column's sniffed/target type — gets a quoted, escaped string literal, letting Firebird implicitly `CAST` it or surfacing a clear per-statement error from `runBatch()`'s existing error isolation). The "create new table" path additionally opens the generated `CREATE TABLE` DDL in an editor first (mirrors `mock-data.ts`'s "generate then open" convention, for review/audit) before running it automatically.
- Wired to `firebird.database.importFlatFile` (same command/menu-registration pattern as the other database-node actions) → `NodeDatabase.importFlatFile()`.
- Not unit-tested itself (VS Code dialog orchestration — same boundary this repo already draws for `node-database.ts`'s other wizard-style methods like `restoreDatabase()`); all its actual logic lives in the already-tested `flat-file-parser.ts`.

### Phase 3 — map onto an existing table

- After parsing, a `QuickPick` asks "Create a new table" or "Map onto an existing table". The existing-table path lists tables via `getTablesQuery(0)`, then reads the chosen table's real columns via `tableInfoQuery()` — the same query the tree's `NodeTable.getChildren()` already uses for its field list, so this reuses live metadata rather than a second schema-fetching path.
- `mapFirebirdFieldToSqlType()` (`flat-file-parser.ts`) converts one `tableInfoQuery()` row into the same `sqlType` vocabulary (`INTEGER`/`BIGINT`/`NUMERIC(p,s)`/`BOOLEAN`/`DATE`/`TIMESTAMP`/`VARCHAR`/`DOUBLE PRECISION`) `inferColumnType()` already produces for the "new table" path — Firebird stores `NUMERIC`/`DECIMAL` as an `INTEGER`/`INT64`/`SMALLINT` column with `RDB$FIELD_SUB_TYPE` 1/2 and a negative `RDB$FIELD_SCALE` rather than its own storage type, so that combination is checked first. Sharing this vocabulary means `cellToSqlLiteral()` (the numeric/boolean-vs-quoted-string literal logic) needed zero changes to support existing-table columns — it's reused verbatim by the new `buildInsertStatementForMapping()`.
- `autoMapColumns()` proposes a default mapping by matching each target column's name against the file's headers (case-insensitive, through the same `sanitizeIdentifier()` normalization column names already go through) — the common case where a CSV's headers already match the target table's columns. Shown to the user as an editable preview (`window.showInformationMessage` with a `detail` listing every `TARGET ← source` pair, plus which file columns go unused) with **Use This Mapping** / **Customize Mapping** buttons; customizing walks each target column via one `QuickPick` (file headers, an explicit "Skip this column" option, and the auto-suggested header — if any — labeled and listed first), letting the user override anything the automatic pass got wrong.
- An unmapped target column is left out of the generated `INSERT` entirely (`buildInsertStatementForMapping()`), not forced to `NULL` — so the table's own default value/nullability rules apply, the same as leaving a column out of a hand-written `INSERT` would.
- **Found and fixed a real, unrelated gap along the way**: `tableInfoQuery()` (`src/shared/queries.ts`) never mapped Firebird's `RDB$FIELD_TYPE` 23 (`BOOLEAN`, introduced in FB 3.0) — it fell through to `'UNKNOWN'`. This affected the tree's own field-type display too (a boolean column showed as `UNKNOWN` there already, before this phase), not just the import wizard's type mapping; fixed by adding the missing `WHEN 23 THEN 'BOOLEAN'` case. New `tableInfoQuery` test coverage locks this in.
- **Verified against a real Firebird 6.0 server** (`/opt/firebird`, a scratch database with one column of every type `mapFirebirdFieldToSqlType()` handles) — and it caught a genuine bug the unit tests, using idealized already-clean strings, had missed entirely: `tableInfoQuery()`'s `CASE` has string-literal branches of different lengths (`'BLOB'` vs `'TIMESTAMP'`, etc.), which Firebird types as a single fixed-width `CHAR` sized to the *longest* branch (`'TIMESTAMP'`, 9 characters) — blank-padding every shorter result to match. `mapFirebirdFieldToSqlType()`'s `switch` was comparing against un-padded strings, so it matched almost nothing except the one longest branch; every other real column silently fell through to the `VARCHAR` (quoted-string) fallback. Functionally this mostly still worked by accident, since Firebird's implicit `CAST` from a quoted string is fairly forgiving for `INSERT`, but it defeated the whole point of reading the target's *real* type, and wasn't something to leave relying on luck. Fixed with a `.trim()` before the `switch` — `node-field.ts`'s tree display already independently trims this exact column for the same reason, so this brought the new code in line with a quirk the codebase already knew about elsewhere. The test suite's `field()` helper now pads `FIELD_TYPE` the same way real data does, so this class of bug can't silently regress by testing against already-clean strings again.

The pre-existing analogs below were the starting point but aren't reused directly — `mock-data.ts`'s INSERT-building/`createSQLTextDocument()` pattern is echoed, but this feature has its own parser rather than depending on Mockaroo or the LLM:

- `src/mock-data/mock-data.ts` — generates rows via the Mockaroo API, builds an `INSERT` via `Firebird.escape()`, opens it with `Driver.createSQLTextDocument()`. Good precedent for the "build INSERT statements, open as SQL" half of this feature, but it doesn't parse *user-supplied* data.
- The `/designSchema` Copilot command (`src/copilot/copilot-chat-participant.ts`) infers a `CREATE TABLE` from pasted sample data via the LLM — useful for schema *inference*, but it's chat-based, one-shot, and doesn't load the resulting data.
- Export already exists in the other direction: `src/result-view/` exports results to CSV/JSON/XLSX/PDF, so there's a reference for CSV formatting conventions/dependencies already in `package.json` worth reusing for parsing.

### Phase 4 — Copilot-assisted type/naming suggestions

- Right after local schema inference, a new "create new table" step (`QuickPick`) asks how to decide column names/types: **Use inferred types** (default-first option, fast, fully local, no Copilot) or **Suggest types with Copilot** — the wizard works identically to before if this is skipped or if Copilot isn't installed, matching the design doc's requirement that Copilot stay optional.
- Follows the same "small structured JSON edit, deterministic code applies it" split already used by the Schema Designer's Ask Copilot panel and the Data API Builder's Copilot-assisted scoping, rather than letting the model produce DDL directly:
  - `buildTypeSuggestionPrompt()` describes each column by index/header/inferred type/nullability plus up to 5 sample rows, and asks for exactly one `{name, sqlType, nullable}` triple per column index, explicitly forbidding adding/removing/reordering columns.
  - `parseSuggestedSchema()` validates the response: invalid JSON or a wrong column count is rejected outright (there's no safe way to reconcile a mismatched count against the file's actual columns), and any entry is applied field-by-field, falling back to the original local inference for anything missing/malformed rather than trusting the model's shape blindly. A suggested name is always re-run through `sanitizeIdentifier()` regardless of what Copilot returned.
  - `suggestTypesWithCopilot()` (not unit-tested — VS Code `lm`/progress orchestration, same boundary drawn elsewhere in this module) calls `vscode.lm.selectChatModels({ vendor: "copilot" })`, and falls back to the unchanged local inference — with a clear notification, never a hard failure — if no model is available, the user cancels, or the request errors.
  - Scoped to the "create new table" path only — an existing table's real column types (phase 3) are fixed and not something this wizard can change, so there's nothing for Copilot to suggest there.
- New unit tests: `src/test/flat-file-import.test.ts` (16 tests) cover `buildTypeSuggestionPrompt()` and `parseSuggestedSchema()`, including markdown-fence stripping, field-by-field fallback, name sanitization, and rejecting wrong/missing column counts.
- **Verified against a real Firebird 6.0 server**: a scratch script parsed a real CSV with a ZIP-code column (`02139`, `94105`, ...) that the local sniffer — correctly, by design — infers as `INTEGER` (would lose the leading zero) and a cryptic `c1` header; simulated a plausible Copilot response (including a markdown fence and one deliberately malformed entry) upgrading `ZIP` to `VARCHAR(5)` and renaming `c1` to `CUSTOMER_NAME`; ran the real `parseSuggestedSchema()` → `buildCreateTableDDL()` → `buildInsertStatement()` pipeline; and actually executed the generated `CREATE TABLE`/`INSERT` statements against the live server. Re-selecting confirmed the leading zero was preserved (`"02139"`) and the renamed column round-tripped correctly. All 12 assertions passed; the scratch database was dropped afterward.

### Large-file streaming (CSV/TSV)

- **`DelimitedRowStreamParser`** (`src/shared/flat-file-parser.ts`, private, exercised via `streamDelimitedRows()`) is a resumable version of the same RFC 4180-ish state machine `parseDelimited()` already used — `parseDelimited()` is now a thin wrapper around it (`feed()` the whole string, then `flush()`), so there's exactly one CSV/TSV parsing implementation to keep correct, not two that could quietly drift apart. `streamDelimitedRows(stream, delimiter)` feeds it successive chunks from a Node `Readable` (e.g. `fs.createReadStream()`) instead of one whole in-memory string, yielding each completed row as soon as it's found.
- Two decisions are genuinely ambiguous exactly at a chunk boundary and can't be resolved until the next chunk (or end-of-input) arrives, so both are deferred via a one-bit "pending" flag rather than guessed at: a `"` as a chunk's very last character while inside a quoted field (a doubled-quote escape vs. the field's real closing quote — resolved by peeking at the very first character of the *next* chunk), and a bare `\r` as a chunk's last character (part of a `\r\n` pair vs. a lone old-Mac-style line ending). `flush()` resolves a still-pending quote as a real close, since there's no more input left for it to have been a doubled `""`.
- A genuinely blank physical line (no delimiter characters at all) parses to a single empty-string cell and is silently skipped by `streamDelimitedRows()`, rather than yielded as a spurious all-NULL row — the practical equivalent of `parseDelimited()`'s whole-file "drop trailing blank lines" pass, without needing to buffer a row's worth of lookahead to tell "trailing" from "not trailing" in a live stream (an already-yielded row can't be un-yielded once the wizard has started inserting it). One disclosed, narrow difference from the whole-file parser: a legitimately blank value in the *middle* of a genuinely single-column file is indistinguishable from a blank line and gets skipped too, rather than inserted as one all-NULL row.
- **The wizard (`src/flat-file-import/index.ts`) now only ever holds a *bounded preview sample* of a CSV/TSV file in memory** (`readDelimitedPreview()`, capped at `PREVIEW_ROW_CAP` = 200 — the same number `inferSchema()`'s own default sample size already used, so there's no value reading further just for a preview) for schema inference, the "map onto an existing table" preview, and the optional Copilot type-suggestion prompt. The actual import (create-table or existing-table INSERTs) re-streams the file from disk via `streamDataRows()`, buffering exactly `INSERT_CHUNK_SIZE` (200) rows at a time before running a batch and discarding them — the file's full row set is never materialized as one array. A small, typical file (fewer rows than the preview cap) is a special case: the preview read already reached end-of-file, so the wizard reuses that already-in-memory sample directly instead of pointlessly re-opening and re-streaming the file a second time — small/typical files behave exactly as before this phase, including the exact "X / Y row(s)" progress percentage and confirmation-dialog row counts; only a file with more rows than the cap switches to the streaming path, where the exact total isn't known upfront (progress instead shows a running "N row(s) imported so far…" count, and confirmation dialogs say "the file's rows" rather than a specific number).
- **Scoped to CSV/TSV only — JSON is a deliberate, disclosed exclusion.** A JSON array of objects doesn't line up with a byte-chunk boundary the way delimited text does (a `"`, `{`, or `}` can appear anywhere, and nesting depth has to be tracked correctly across chunks), so a *correct* streaming JSON parser is a meaningfully bigger, riskier lift than delimited-text streaming turned out to be. A huge JSON file still reads entirely into memory, exactly as it always has — see "Explicitly deferred" below.
- **Verified two ways.** Unit tests (`src/test/flat-file-parser.test.ts`) feed the streaming parser pre-split synthetic chunks at the exact byte offsets hardest to get right — mid-field, mid-quote, a doubled-quote escape split exactly at the boundary, `\r\n` split between the two characters, a file ending exactly on a quoted field's closing quote — plus a test asserting `streamDelimitedRows()` produces the *exact same result* as `parseDelimited()` for the same input, whole or chunked. A second suite drives the new file-reading helpers (`detectDelimiterFromFile()`, `readDelimitedPreview()`, `streamDataRows()`) against real temp files on disk, including a 3000-row real file (comfortably past both the preview cap and a real `fs` stream's default chunk size) confirming every row arrives in order with none lost or duplicated, and a quoted multi-line field deliberately positioned to span a real internal read-chunk boundary. Then, live against a real Firebird 6.0 server: a genuine 5000-row, ~165 KB CSV (with quoted fields scattered throughout, not just conveniently near the start) was run through the exact preview → infer → stream → insert sequence the wizard itself follows, actually executing the generated `CREATE TABLE`/`INSERT` statements — confirming `readDelimitedPreview()` correctly reports `truncated: true`, every one of the 5000 rows landed with no gaps/duplicates (`MIN(ID)`/`MAX(ID)`/`COUNT(DISTINCT ID)` all checked), a quoted field with an embedded comma round-tripped correctly, the very last row in the file made it in (proving the stream ran to true end-of-file, not just to the preview cap), and — the actual point of this phase — the in-memory row buffer never exceeded one chunk's worth (200) of rows at any point during the import.

### Explicitly deferred (not done)

- **JSON large-file streaming**: only CSV/TSV got a streaming path this pass — see above for why. A multi-GB JSON array-of-objects file still reads entirely into memory.

## Proposed feature

A guided multi-step wizard (VS Code `QuickPick`/`InputBox` steps, or a small webview if the column-mapping step needs a table UI):

1. **Pick a file** — CSV, TSV, or JSON (array of objects). Sniff delimiter and whether the first row is a header.
2. **Preview + infer types** — parse the first ~100 rows client-side, infer a Firebird column type per column (INTEGER/BIGINT/NUMERIC/VARCHAR/DATE/TIMESTAMP/BOOLEAN), similar logic to what `/designSchema` currently delegates entirely to the LLM — for the wizard this should be a **local, deterministic** type-sniffer (no LLM round-trip needed for a mechanical CSV-to-DDL inference; save the LLM path for cases the user explicitly wants smarter naming/typing suggestions, and let the wizard optionally call the same Copilot prompt as a "suggest types" button rather than a hard dependency, since flat-file import must work without Copilot installed).
3. **Choose target** — create a new table (using the inferred DDL, editable before running) or map columns onto an existing table (reuse tree metadata already available to `NodeTable`).
4. **Load** — batch-insert the parsed rows via `Driver.runBatch()` (chunked, since Firebird has statement/parameter limits and large CSVs shouldn't be one giant multi-statement string) with a progress indicator (`vscode.window.withProgress`).

## Technical notes

- New pure module `src/shared/flat-file-parser.ts` (CSV/TSV/JSON parsing + type sniffing) — unit-testable like `sql-formatter.ts`, and it's the piece most worth getting right since it's shared with any future "paste sample data" improvements to `/designSchema`.
- No CSV parsing dependency currently exists in `package.json` — decide whether to hand-roll a minimal RFC 4180 CSV parser (repo has a stated preference for avoiding unnecessary abstractions/dependencies) or pull in a small, well-maintained library; either way keep it isolated behind the parser module so the choice is swappable.
- Large-file handling: stream/chunk rather than loading an entire multi-GB CSV into memory; cap the preview step's sample size regardless of file size.
- Reuse `Firebird.escape()`/parameterized inserts (whichever `NodeClient`/`NativeClient` already prefers for `runBatch`) rather than hand-building escaped SQL strings, to avoid injection/quoting bugs with arbitrary user data (dates, embedded quotes, unicode).
