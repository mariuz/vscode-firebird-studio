# Backup/Restore: Expose gbak Options

**Inspired by**: [vscode-pgsql](https://github.com/microsoft/vscode-pgsql)'s Database Backup (Preview, 1.27.0) — "Choose Backup... from a server or database in Object Explorer to create a `pg_dump` backup. Configure the output format, schema/data scope, object filters, compression, parallel jobs, and advanced options, then monitor or cancel the backup from VS Code."

## Current state in Firebird Studio

**Phase 1 is done.** `NodeDatabase.backupDatabase()`/`.restoreDatabase()` (`src/nodes/node-database.ts`) already run a real `gbak` subprocess with real progress reporting — a `showSaveDialog`/`showOpenDialog` for the file, a status-bar spinner, and (since `docs/roadmap/connection-management-enhancements.md`'s phase 4) a Background Tasks entry.

- **Phase 1 — backup options — done.** `backupDatabase()` now shows a multi-select QuickPick (`BACKUP_OPTION_ITEMS`) before the file picker, offering the four flags below; leaving everything unchecked (or pressing Enter immediately) produces the exact same `args` array as before this phase — Escape/dismissed cancels the whole backup, matching the file picker's own existing cancel behavior. New `src/shared/gbak-options.ts`'s pure `buildBackupFlags(choices)` turns the picked checkboxes into the actual `-G`/`-ZIP`/`-M`/`-NT` flags, kept separate from the QuickPick/child-process code specifically so it's unit-testable without a real file dialog or subprocess. **All four flags verified directly against a real `gbak` (Firebird 6.0) run against the seeded test database** — `-G` (no size difference, as expected — garbage collection doesn't change backup content), `-M` (87552 vs 101376 bytes, correctly schema-only), `-ZIP` (11776 bytes, real compression), `-NT` (100352 bytes, smaller than the transportable default), and all four combined together (9728 bytes, exit code 0) — not just that the flags exist in `gbak -z`'s help text, but that Firebird's real `gbak` actually accepts and honors each one.
- **A genuine, separate pre-existing gap found and fixed while verifying phase 1 live**: `gbak` turned out to not be on `PATH` in this session's own sandbox at all (confirmed via `which gbak`) — only reachable at `/opt/firebird/bin/gbak` — yet `cp.execFile("gbak", args)` called a bare, unconfigurable binary name, unlike `isql`'s own `firebird.isqlPath` setting/`resolveIsqlExecutable()`. Fixed with a new `firebird.gbakPath` setting and `resolveGbakExecutable()`/`gbakCandidates()` (`gbak-options.ts`), mirroring `resolveIsqlExecutable()`/`resolveDockerExecutable()` exactly — this codebase already keeps these three small and independent rather than sharing one abstraction between them, so a third near-identical copy was written the same way rather than introducing a new shared helper the codebase hasn't asked for. **One real wrinkle**: unlike isql's own `-z` (exits cleanly), a real `gbak -z` always exits non-zero — confirmed live, it still prints its version banner to stdout ("gbak:gbak version...") *then* errors with "requires both input and output filenames" since `-z` alone isn't a complete backup/restore command — so `extension.ts`'s new `checkGbakExecutable()` checks stdout for the version banner instead of the exit code, unlike `checkIsqlExecutable()`'s exit-code check. Both `backupDatabase()`/`restoreDatabase()` now take the resolved executable path as a parameter (defaulting to `"gbak"` for any caller that doesn't resolve one first) rather than hardcoding it. Verified against the real sandbox's actual broken state: unresolvable with no setting (matches reality), resolves correctly once `firebird.gbakPath` points at the real binary, and correctly refused for a bogus path.
- Not done yet: restore options (phase 2), cancel support (phase 3), parallel/multi-file backup (phase 4).

### Testing

`src/test/gbak-options.test.ts` (14 tests) covers `buildBackupFlags()` (no flags produce `[]`, each flag maps to its correct switch, multiple choices combine correctly), `gbakCandidates()`, and `resolveGbakExecutable()` (working custom path, failing custom path doesn't fall back to PATH search, PATH-search success/failure, empty-string custom path treated as unset) — the last suite mirrors `isql-terminal.test.ts`'s own `resolveIsqlExecutable()` tests exactly. The QuickPick/subprocess wiring itself isn't independently tested, matching this repo's established boundary for VS Code dialog orchestration — but was verified live (see above) three separate ways: the individual and combined `gbak` flags run directly against the seeded test database, and `resolveGbakExecutable()` run against this sandbox's real, genuinely-broken "gbak not on PATH" state plus a real custom-path override pointing at `/opt/firebird/bin/gbak`.

## Proposed feature

`gbak` itself already supports the switches that would matter here — this is exposing existing `gbak` capability through UI, not building new backup logic. Verified directly against a real `gbak -z`'s own help output (Firebird 6.0) rather than trusted from memory — several of the flag letters originally guessed here turned out wrong (`-fa` is actually "blocking factor", not metadata-only; `-k` is "no shadows", not "no validity checks"):

- **Backup options** (`gbak -b` flags): `-G` (inhibit garbage collection, faster backup), `-NT` (non-transportable format — Firebird's own default *is* transportable, `-NT` opts out for a smaller/faster same-platform-only backup), `-M` (metadata only, no data), `-ZIP` (zip-compressed backup file), `-SE` (use the Services API instead of a local file-path connection, needed for a remote server without local `gbak` access to the file path). `-PAR n` (parallel workers, Firebird 4+) for large databases.
- **Restore options** (`gbak -c`/`-r`/`-rep` flags): `-P <size>` (override page size), `-O` (restore one table at a time, useful for diagnosing a restore failure), `-N` (skip restoring validity/`NOT NULL`/`CHECK` conditions), `-K` (restore without creating shadow files). Note `-REP`/`-R(ECREATE) OVERWRITE` are *top-level* restore-mode switches (used instead of `-C`, not alongside it) for replacing an already-existing database, not modifier flags — today's `restoreDatabase()` always uses `-C`, which fails outright if the target file already exists; offering "replace an existing database" needs switching the top-level flag, not adding one.
- A QuickPick-based "Backup options" step (matching this codebase's existing wizard patterns, e.g. `connection-wizard.ts`) before the file picker, defaulting to today's exact behavior (no flags — Firebird's own defaults) so nothing changes for anyone who doesn't open it.
- **Cancel support**: `gbak` is a real child process (`cp.execFile`) already — a `Cancel` button on the status bar item or the Background Tasks entry could call `child.kill()`, not currently offered at all.

1. ~~A backup-options QuickPick (garbage collection `-G`, compression `-ZIP`, metadata-only `-M`, non-transportable `-NT`) threaded into `backupDatabase()`'s existing `args` array.~~ — **done**, see above.
2. Restore-options QuickPick (page size `-P`, one-at-a-time `-O`, skip validity `-N`, no shadows `-K`) for `restoreDatabase()`; a separate "replace an existing target" choice that switches the top-level `-C`/`-REP` flag rather than adding a modifier.
